# 采用「上传即复制URL」的图片交互模式

_来源：976215b → 1d42e71 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
在自由画布（HtmlMode）中集成图片上传功能时，需要决定上传成功后如何处理返回的图片 URL，以适配项目「离线 AI 辅助生成 HTML」的核心工作流。

## 决策驱动
- 与非技术用户的工作流匹配度
- 避免自动代码修改带来的光标/上下文风险
- 保持离线 AI 交互的一致性

## 备选方案
- **上传后自动将 <img> 标签插入编辑器 HTML** _（已否决）_ — 优点：传统编辑器的直观体验，一步到位；缺点：非技术用户难以控制插入位置（光标状态复杂）；与「先发给 AI 再生成代码」的离线工作流不匹配
- **上传后复制 URL 到剪贴板并提示用户发送给 AI** — 优点：完美契合「用户->AI->HTML」的闭环；无需处理编辑器内部的光标和 DOM 操作逻辑；UX 简单明确；缺点：多了一步手动粘贴的操作

## 决策
选择「上传 → 复制 URL → 用户发给 AI」方案。在 `src/modes/html/HtmlMode.tsx` 中实现上传按钮，成功后调用 `copyText(url)` 并弹出 Toast 提示，不自动修改编辑器内的 HTML 代码。

## 影响
简化了前端逻辑，避免了复杂的富文本光标管理问题。但要求用户必须理解并执行「将 URL 提供给 AI」这一步骤，因此需要配合用户引导（UserGuidePopover）和清晰的 Toast 提示。
