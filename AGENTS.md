# AGENTS.md

本文件为仓库级 AI 代理协作说明，适用于 Codex、Antigravity、Windsurf、Cursor、Claude 及其他兼容 `AGENTS.md` 的工具。

## 0. 行为准则 (Behavioral Guidelines)

本准则旨在减少 AI 在复杂任务执行中常见的偏差和错误。请根据具体任务的要求灵活结合使用。

**权衡说明**：本准则倾向于**"谨慎优先于速度"**。对于极其简单的琐碎任务，请自行判断并简化流程。

### 0.1 思考先行 (Think Before Executing)
**不作假设。不掩饰困惑。揭示取舍。**

在开始执行任何任务之前：
* **明确陈述你的假设**。如果不确定，主动提问。
* 如果对我的需求有多种理解方式，**把它们列出来——不要默默替我做决定**。
* 如果存在更简单、更直接的解决方案，明确提出来。必要时，请提出合理的反对意见（Push back）。
* 如果某处信息不清晰，立即停止。明确指出哪里让你感到困惑，并向我提问。

### 0.2 简单优先 (Simplicity First)
**用解决问题的最小输出。拒绝任何猜测性的发散。**

* 不做需求之外的任何附加动作。
* 不做仅为单次任务而设计的过度抽象或复杂框架。
* 不提供未经要求的"灵活性"或"全局适用性"。
* 不去应对几乎不可能发生的极端边缘情况。
* 如果你输出了 2000 字，但其实 500 字就能说清，**重写它**。

**自问自答**："一位资深专家或管理者会认为这个方案/这份文档太复杂了吗？"如果是，请简化。

### 0.3 手术刀式执行 (Surgical Changes)
**只触碰你必须改动的部分。只清理你自己制造的烂摊子。**

当你在修改或编辑现有的文档、方案或数据时：
* 不要顺手"优化"相邻的段落、逻辑或排版。
* **不要重构没有坏掉的东西**，除非用户明确要求。
* 保持现有的风格和语气，即使你个人更倾向于另一种表达方式。
* 如果你注意到了其他无关的错误或废话或其它应该进行的优化，你可以提一句——但**不要擅自修改**。

当你的修改导致了上下文割裂时：
* 只清理因为**你的**改动而产生的冗余信息（例如不再适用的上下文或废弃的前提）。
* 除非被要求，否则不要删除原有的历史遗留信息。

**核心检验标准**：你修改的每一句话、每一个数据，都必须能直接追溯到用户的当前需求。

### 0.4 目标驱动执行 (Goal-Driven Execution)
**定义成功标准。循环迭代直到验证通过。**

将模糊的任务转化为可验证的具体目标：
* "优化这篇报告" → *检查逻辑漏洞，修正错别字，并确保每个核心论点都有数据支撑。*
* "分析这组数据" → *找出异常值，并计算出核心指标的环比增长。*
* "调整这个方案" → *确保调整后的方案预算不超过原定上限。*

对于多步骤的复杂任务，先列出一个简短的执行计划：
```text
1. [执行步骤] → 验证: [检查项是否达标]
2. [执行步骤] → 验证: [检查项是否达标]
3. [执行步骤] → 验证: [检查项是否达标]
```
强健的"成功标准"能让你自主闭环完成工作；而软弱的"成功标准"（比如"把它弄好"）只会导致不断的澄清和返工。

## 1. 项目语言与文档规范

本仓库默认使用中文进行规划、说明和协作记录。

* 需要建立明确的规划文档、需求说明、流程说明、复盘说明等文档，并且优先使用中文。
* 面向项目维护者或后续 AI 代理的说明文档，优先使用中文。
* 代码中的必要注释使用中文；仅在引用库 API、协议字段、命令参数或英文原文时保留英文。
* 代码本身的变量名、函数名、文件名按技术栈惯例处理，不强行中文化。
* Git的时候，也要使用中文说明。

## 2. 当前项目总目标

本项目（markdown2view）的总目标是：构建一个**纯前端、零后端**的「Markdown / HTML 多场景渲染与导出工作台」，把同一份内容渲染为面向不同受众的成品形态（公众号文章 / A4 正式文档 / 社交媒体卡片 / AI 生成的风格化 HTML），并一键复制或导出（富文本 / 图片 / PDF）。

后续代理继续工作前，应先阅读：

* 总体实施规划（技术架构）：`docs/技术架构设计.md`
* 技术路线图：`docs/技术路线图.md`
* 设计指引：`DESIGN.md`
* 项目使用 Skill：`.agent/skills/`（含 `awesome-design-md` 设计风格、`design-system`、`frontend-dev`、`vercel-react-best-practices`）
* 参考项目：`public/vendor/r-markdown/`（公众号渲染引擎）、`public/vendor/html-anything/`（HTML 沙箱容器与导出层）

核心要求：

* 技术栈：React 18 + Vite + TypeScript + Tailwind；状态用 Zustand；编辑器用 CodeMirror 6。
* AI 协作为「复制指令 → 外部 AI 生成 → 回填系统」的离线工作流，**不集成本地 CLI、不需要后端**。
* 公众号渲染引擎从 `public/vendor/r-markdown` 原样移植（其引擎为框架无关的纯 TS）。
* 导出能力（iframe→PNG 等）从 `public/vendor/html-anything/next/src/lib/export/` 移植。
* 前端项目依赖统一记录在 `package.json`；不涉及 Python，故不创建 `requirements.txt`。

## 3. 执行经验与开发规范

### 3.0 模块复用优先原则（Reuse First）

本项目的一个核心架构原则是**"首先复用项目中已有的模块，绝不就地定义重复资源"**。所有 AI Agent 在进行任何代码变更前，必须按以下顺序检查。

#### 第一步：确认项目已有什么

在动手写任何代码之前，先全面扫描以下位置，确认是否已有可复用的模块：

| 资源类型 | 集中管理位置 | 说明 |
|----------|-------------|------|
| 字体配置 | `src/lib/fonts.ts` | 统一管理宋体/仿宋/黑体，不在业务代码中直接定义字体名（如 `SimSun`） |
| 风格提示词 | `src/data/designPrompts/styles/` | 风格模板和模板生成提示词 |
| 图标 | `src/components/ui/Icon.tsx` | 所有通用 UI 图标 |
| 导出管线 | `src/lib/exportPdf.ts` / `exportImage.ts` / `exportDocx.ts` / `exportPpt.ts` | 复用现有的逐页截图/PDF/DOCX/PPT 导出函数 |
| 组件库 | `src/components/ui/` | 按钮、卡片等 UI 组件 |
| 导出工具 | `src/lib/use*` hooks 及 `useExportAction.ts` | 导出动作的统一封装 |
| 工具函数 | `src/lib/` 下各独立模块 | 不在业务代码中就地复制粘贴 |

#### 第二步：如非必要不增加冗余

当你想要在某业务文件（如 `exportPptEditable.ts`）中定义某个资源列表或配置时，先自问：

1. 这个东西是否已经存在于 `src/lib/` 的某个模块中？
2. 如果不存在，它是否真的仅属于当前文件的内部实现细节？
3. 如果是通用资源（字体、颜色、尺寸、正则白名单等），应该抽象到公共模块而不是就地定义。

#### 第三步：如需扩展在原模块上加

当现有模块需要新功能时（如 fonts.ts 需要新增一种字体风格），在原模块添加新的 `FontFamilyOption`，而不是在下游业务文件中就地加入新字体名。

#### 第四步：跨文件重复时提取为独立模块

如果某段逻辑（如文件名校验、abort 取消机制、scaleX/scaleY 尺寸换算）在多个文件中重复出现，应提取为独立模块供所有文件复用。

#### 具体案例：pptxgenjs 中文支持

pptxgenjs 等第三方库生成文档时，中文文本需要指定支持中文的字体。

这是一个典型的**"就地定义"陷阱**：很容易在业务文件中写 `const chineseFonts = ['SimSun', 'Microsoft YaHei', ...]` 来解决。

正确做法：从 `src/lib/fonts.ts` 获取已有字体配置，中文文本统一使用 `SimSun`。这样当字体配置需要修改时，只需改动一个文件。

* **[开发经验与偏好记录](docs/开发经验与偏好记录.md)**：包含用户审美偏好、踩坑经验（如隐藏 DOM 测量、公众号引擎排版规范、编辑器优化等）。

### 3.1 Git 提交与验收要求

每次完成较大的功能改进或 bug 修复后，必须及时进行 Git 提交，并严格遵守代码与提交规范。

**提交前验收检查**：
在执行 `git commit` 前，必须确保通过以下验收检查（详见 `docs/代码与提交规范.md`）：
1. `pnpm typecheck`：通过 TypeScript 静态类型检查。
2. `pnpm test`：通过全部的 Vitest 单元测试。
3. `pnpm build`：确保构建不报错，并观察终端输出的打包产物体积，确保核心 chunk 没有异常膨胀。

**提交规范**：
- 遵循 Angular 规范的提交信息格式：`<type>(<scope>): <subject>` + `<body>`
- Type 包括：feat、fix、docs、style、refactor、perf、test、build、chore
- 提交信息使用中文，清晰描述改动内容和原因

**执行要求**：
- 每完成一个独立的功能模块或 bug 修复，应立即提交，避免累积过多改动导致提交信息模糊
- 提交前必须先执行验收检查，确保代码质量
- 不得跳过验收检查直接提交

### 3.2 指令-渲染规则-用户操作-外部AI 同步原则

本项目是「渲染规则 → AI 指令 → 用户操作 → 外部 AI」的闭环体系：

1. **渲染规则**：代码层面对输入内容的真实解析、排版、安全净化与导出约束（如 HTML 黑名单、Markdown 自定义标签、分页容器、外链资源限制等）。
2. **AI 指令**：通过 `src/lib/aiGuide.ts`、`src/data/designPrompts/utils.ts` 等生成的提示词，用户复制后发给外部 AI。
3. **用户操作**：用户把外部 AI 生成的 Markdown / HTML 贴回编辑器。
4. **外部 AI**：接收指令并生成内容，它只能「看到」指令，看不到代码实现。

**因此，任何对渲染规则的修改，都必须同步到对应 AI 指令中**；否则外部 AI 会持续输出与代码层不匹配的内容，导致用户得到“AI 生成正常，但系统渲染异常、样式丢失或语义错误”的体验割裂。不同步的典型表现包括：使用被禁止的标签/属性（如 `<script>`）、错误使用标志符号（如 `<page-break/>`）、忽略强制分页容器（如 `<section class="slide">`）、误解样式效果（如缩放、安全区、比例）等。

执行要求：
* 修改渲染规则（新增/删除/变更标签、属性、CSS、安全策略、分页容器、标志符号、外链规则、样式语义等）时，**必须同时检查并更新对应模式的 AI 指令**。
* 新增安全限制（如 XSS 净化黑名单）时，**必须在指令中明确告诉外部 AI 哪些能力被禁止、推荐用什么替代方案**。
* 不要把实现细节原封不动地搬进指令，但要保证指令对外部 AI 足够清晰、全面、可执行，让它能稳定产出符合当前渲染规则的内容。
* 完成规则修改后，应做一轮「指令对照检查」：逐条核对指令描述与实际代码行为是否一致，发现不一致立即修正。

其他通用环境要求：
* PowerShell 环境下按 PowerShell 语法执行命令。
* `r-markdown` 的渲染引擎是纯 TypeScript（返回 HTML 字符串），已整体移植到 `src/engine/`。
