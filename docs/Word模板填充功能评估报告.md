> 本报告承接 `docs/A4文档Word导出调研与规划.md` 中已有的调研结论，
> 针对**「指定 Word 模板 → 根据模板填充内容」**完整功能链路开展新增评估。
> 评估日期：2026-06-17
>
> **📌 决策修正（2026-06-17）**：经讨论确认，本功能不另起模板引擎，

> AI 指令采用指令库形式（对标自由画布 `PromptLibrary`），

## 执行摘要

| 维度 | 结论 |
|------|------|
| **可行性** | 技术上可行，前端零后端约束下有完整实现路径 |
| **推荐实现方向** | 自研模板体系（JSON Schema 驱动），**不依赖 .docx 模板文件** |
| **优先级建议** | 中低优先级，建议在 Word 基础导出（Task 1-8）完成后作为独立功能迭代 |
| **主要风险** | .docx 模板解析复杂度高；AI 多字段协同填充的 Prompt 工程难度中等 |
| **最终决策** | 复用现有自定义标签体系 + 指令库模式，封面页作为独立指令变体（见附录） |

## 一、产品交互设计评估

### 1.1 用户选择模板的交互流程

**推荐方案：模板库面板（侧边抽屉 / 模态卡片网格）**

```
入口：A4 文档模式工具栏 → "从模板开始" 按钮
  ↓
模板选择面板（模态框或左侧面板）
  ├─ 分类标签：合同/报告/简历/通知/协议/其他
  ├─ 模板卡片：缩略图 + 名称 + 字段数量标注
  └─ 搜索框
  ↓
选中模板后：
  ├─ 右侧预览区域展示模板骨架（含高亮占位符）
  └─ 下方「填充字段」区域展开字段表单
```

**交互注意点：**

- 与现有工具栏设计语言保持一致（参考 `开发经验与偏好记录.md` § 2.8）——工具栏操作模块化，不堆叠按钮。
- 模板预览采用缩略图（参考风格指令缩略图方案 § 2.10），`scale(0.8)` 轻度缩放 + 放大初始字号，确保关键字段清晰可读。
- 「从模板开始」与「自由编辑」并列，互不干扰，用户可随时切换至纯 Markdown 编辑。

### 1.2 模板预览与字段标注方式

**两种实现方案对比：**

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| A：静态 HTML 骨架预览 + 高亮占位符 | 用 HTML/CSS 渲染模板外观，占位符用彩色方块标注 | 实现简单，可用现有渲染引擎 | 与真实 Word 样式有视觉差距 |
| B：真实 .docx 解析 + 注释层叠加 | mammoth.js 将 .docx 转 HTML，再叠加字段高亮层 | 预览保真度高 | mammoth.js 无法完美还原 Word 样式，中文字体尤其差 |

**推荐方案 A：静态 HTML 骨架预览**

原因：项目已有 A4 Paged.js 渲染管线，可直接复用；自研模板体系中占位符本身就是 Markdown 或结构化变量，不需要从 .docx 中反向解析。字段高亮使用带颜色边框的 `inline-block` 标注，点击跳转到表单对应字段。

### 1.3 内容填充的编辑体验

**字段映射 UI 设计（两种模式并行）：**

**模式一：表单填充（结构化）**

```
┌─────────────────────────────────────────────────────┐
│ 模板：劳动合同                                       │
├──────────────┬──────────────────────────────────────┤
│ 甲方（雇主）  │ [________________]                   │
│ 乙方（员工）  │ [________________]                   │
│ 职位          │ [________________]                   │
│ 入职日期      │ [日期选择器]                         │
│ 薪资          │ [________________] 元/月             │
│ 合同期限      │ [________________] 年                │
├──────────────┴──────────────────────────────────────┤
│ [🤖 AI 智能填充]   [清空]   [预览效果]   [导出 Word] │
└─────────────────────────────────────────────────────┘
```

**模式二：Markdown 自由编辑（与现有编辑器融合）**

- 模板展开为带占位符注释的 Markdown 文本，如 `{{甲方公司名称}}` 直接在编辑器中可见
- 用户可直接编辑 Markdown 替换占位符，也可用 AI 批量生成
- 保持项目一贯的「编辑器优先」体验

**推荐：两种模式均支持，表单模式作为低门槛入口，Markdown 编辑作为高级路径**

## 二、AI 指令设计评估

### 2.1 核心 Prompt 设计思路

模板填充场景下的 AI 指令需要解决三个层次的问题：

**层次一：结构理解（让 AI 明白模板是什么）**

```
你是一位专业的文档撰写助手。以下是一份「{{模板名称}}」的结构模板，
其中 {{字段名}} 格式的占位符代表需要填写的信息。

模板结构：
---
{{模板完整骨架文本，占位符保留}}
---
```

**层次二：内容生成（让 AI 根据上下文填充）**

分两种触发方式：

- **有上下文信息时**（用户在聊天框/编辑器中已提供背景信息）：
  ```
  根据以下背景信息，将上述模板中所有占位符填充为具体内容。
  背景信息：{{用户提供的原始文字}}
  
  要求：
  1. 直接输出填充后的完整 Markdown 文本，不加任何解释
  2. 保持原有格式结构不变
  3. 如信息不足，用「[待填写]」代替该字段
  ```

- **无上下文时（AI 自由生成示例）**：
  ```
  请为这份{{模板名称}}生成一套完整的示例填充内容。
  使用符合中国法律法规和商业惯例的典型内容。
  直接输出填充后的 Markdown，不加解释。
  ```

**层次三：多字段协同（避免字段间矛盾）**

关键设计：**批量填充，而非逐字段填充**。

```
注意字段间的逻辑一致性：
- 合同甲方与乙方不能是同一主体
- 日期字段需遵循时间先后顺序（签订日 < 入职日 < 合同到期日）
- 金额字段应与级别/职位相匹配
- 地址字段中的城市应与公司所在地一致
```

### 2.2 AI 工作流设计（符合项目离线工作流）

项目的 AI 工作流是「复制指令 → 外部 AI → 回填系统」，模板填充功能应遵循此约定：

```
[生成指令] 按钮
  ↓ 根据选中模板 + 用户已填字段，动态生成完整 Prompt
[复制到剪贴板]
  ↓ 用户在外部 AI（ChatGPT/Claude/Kimi 等）中执行
[粘贴结果] 区域
  ↓ 系统检测粘贴内容是否包含完整的 Markdown 占位符填充结果
[一键应用到文档]
```

这个流程与现有「风格指令库」的工作模式完全一致，零额外学习成本。

### 2.3 指令设计的潜在风险

| 风险 | 描述 | 缓解策略 |
|------|------|---------|
| 占位符被 AI 误删 | AI 可能忽略某些占位符 | Prompt 中明确要求「保留未填充字段为 [待填写]」并输出完整文本 |
| 输出格式不稳定 | AI 可能输出 JSON / 表格而非 Markdown | 在 Prompt 末尾加「仅输出 Markdown，不加代码块包裹，不加解释」 |
| 多轮对话状态丢失 | 离线工作流无法多轮纠错 | 提供「字段提示」——显示哪些字段仍为 [待填写]，引导用户二次补填 |

## 三、模板解析与渲染评估

### 3.1 模板文件的解析方案选型

**关键决策：是否使用真实 .docx 文件作为模板源？**

**方案 A：以 .docx 文件为模板（解析路径）**

| 工具 | 能力 | 问题 |
|------|------|------|
| docxtemplater | 最专业的 .docx 模板引擎，支持 `{field}` 语法 | 图片/HTML 模块需付费（500€/年），已在原调研中否决 |
| docx-templates | 开源替代品，支持 JS 表达式 | 缺乏 sections/headers/footers/TOC，对复杂格式支持不足 |
| mammoth.js | .docx → HTML 转换 | 仅用于阅读，不支持将填充后的内容写回 .docx |

**结论：.docx 解析路径代价过高**——解析 .docx XML、识别自定义 XML 属性/内容控件/书签等占位符格式，需要深入处理 OOXML 规范，而且免费工具都存在明显短板。

---

**方案 B：自研模板体系（JSON Schema + Markdown 骨架）**

这是**推荐方案**，理由如下：

1. **与项目现有数据模型天然契合**：`DocumentBlock[]` 已经是结构化模型，模板只是预设了 blocks 的初始内容
2. **零额外依赖**：复用现有的 Markdown → DocumentBlock → docx 管线
3. **完全可控**：占位符格式、字段类型、渲染逻辑完全由项目自定义

**模板数据结构设计：**

```typescript
// 模板定义
interface DocumentTemplate {
  id: string
  name: string               // 如「劳动合同」
  category: string           // 如「合同」
  description: string
  thumbnail?: string         // Base64 缩略图或 SVG 骨架
  fields: TemplateField[]    // 字段定义
  markdownTemplate: string   // 含 {{field_id}} 占位符的 Markdown 骨架
  defaultSettings?: Partial<DocumentSettings>  // 推荐页面设置
}

// 字段定义
interface TemplateField {
  id: string                 // 对应 markdownTemplate 中的 {{id}}
  label: string              // 展示名称，如「甲方公司名称」
  type: 'text' | 'date' | 'number' | 'textarea' | 'select'
  placeholder?: string       // 表单输入提示
  required?: boolean
  options?: string[]         // select 类型的选项
  defaultValue?: string
}

// 填充数据
type TemplateValues = Record<string, string>
```

### 3.2 占位符/变量定义规范

**占位符格式：`{{field_id}}`**

与 docxtemplater、Handlebars、Liquid 等主流模板语法保持一致，降低 AI 生成时的认知负担。

规范约定：
- 纯字段替换：`{{company_name}}`
- 带格式提示的字段：`{{contract_date|date}}` （可选，用于表单类型推断）
- 多行文本字段：`{{job_description|textarea}}` （textarea 类型会在表单中展示大文本框）

**模板骨架示例（劳动合同片段）：**

```markdown
# 劳动合同

**甲方（用人单位）**：{{employer_name}}

**乙方（劳动者）**：{{employee_name}}

## 第一条 劳动合同期限

本合同为{{contract_type}}劳动合同，自 {{start_date}} 起至 {{end_date}} 止。

## 第二条 工作内容和工作地点

乙方同意甲方的工作安排，担任 **{{position}}** 岗位工作，
工作地点为 {{work_location}}。
```

### 3.3 填充后的实时预览渲染机制

**填充流程（纯前端，实时）：**

```typescript
function renderTemplate(template: string, values: TemplateValues): string {
  return template.replace(/\{\{(\w+)(?:\|[\w]+)?\}\}/g, (_, key) => {
    return values[key] ?? `{{${key}}}`  // 未填充的保持占位符可见
  })
}
```

- 表单字段 `onChange` → 实时调用 `renderTemplate` → 更新 Markdown 编辑器内容 → 触发 Paged.js 重渲染
- 空字段以带样式的 `[{{field_id}}]` 形式在预览中高亮显示（橙色边框），提醒用户尚未填写
- 全部字段填写完毕后，「导出 Word」按钮状态变为高亮

### 3.4 最终导出为 A4 Word 文档的实现路径

```
TemplateValues + DocumentTemplate.markdownTemplate
    ↓ renderTemplate()（占位符替换）
Markdown 字符串（与现有文档模式完全一致的格式）
    ↓ splitMarkdownBlocks()（已有）
DocumentBlock[]
    ↓ exportDocx()（参照原调研 Task 1-8 实现）
.docx Blob → 触发下载
```

**关键优势**：模板填充后的导出路径与「普通 Markdown 导出 Word」完全共享同一套代码，模板功能本质上是「带预设内容的 Markdown 初始化器」，不引入任何新的导出逻辑。

## 四、结合本项目实际：可行性与优先级评估

### 4.1 开发可行性

| 子功能 | 依赖现有能力 | 新增工作量 | 可行性 |
|--------|------------|-----------|--------|
| 模板数据结构定义 + 预置模板编写 | 无依赖 | 小（JSON + Markdown 骨架） | ✅ 高 |
| 模板选择 UI（卡片网格 + 分类） | 复用风格库 UI 模式 | 小-中 | ✅ 高 |
| 占位符替换引擎 | 无依赖，纯正则 | 极小 | ✅ 高 |
| 表单字段填充 UI | 无依赖 | 中（含表单组件） | ✅ 高 |
| 实时预览（Paged.js 渲染） | 完全复用现有渲染管线 | 极小 | ✅ 高 |
| AI 指令生成 + 复制 | 复用现有 Prompt 系统 | 小 | ✅ 高 |
| 最终导出 Word | **依赖 Word 导出功能（Task 1-8）先完成** | 极小（调用已有接口） | ⚠️ 依赖项 |
| 自定义模板上传（用户上传 .docx） | 需要 docx 解析，复杂度高 | 大 | ❌ 不建议 |

### 4.2 潜在技术风险

**风险一：Word 基础导出尚未实现（最关键依赖）**

原调研规划（Task 1-8）的 Word 导出功能尚在规划阶段。模板填充功能的最终输出依赖该功能。建议严格遵守原调研的「先完成基础通路，再扩展模板功能」的迭代顺序。

**风险二：Paged.js 的实时渲染性能**

表单填充时，每次字段变动都会触发 Markdown 更新，进而触发 Paged.js 重排。对于长文档模板（如合同、报告），重排耗时可能在 500ms-2000ms，用户体验不佳。

**缓解策略**：
- 表单填充阶段使用「快速预览」（纯 HTML 渲染，不经过 Paged.js），仅在用户主动点击「预览/导出」时才触发完整 Paged.js 渲染
- 对表单输入防抖（300ms），避免每次按键都触发重渲染

**风险三：内置模板数量与质量**

模板价值很大程度上取决于内置模板的质量和覆盖场景。初期内置模板不足会使功能体验打折。建议首发至少覆盖 4-6 个高频场景（合同、报告、通知、简历、协议、申请书）。

**风险四：AI 输出的格式一致性**

由于是离线工作流（用户复制到外部 AI），无法控制 AI 输出的格式。不同 AI 对「直接输出 Markdown」的理解不一致，可能输出代码块包裹的 Markdown 或夹杂解释文字。

**缓解策略**：提供智能「提取 Markdown」功能——粘贴区域自动检测并提取代码块内的 Markdown 内容，以及自动剥除非 Markdown 的前后缀文字。

### 4.3 与现有功能的协同效应

| 协同点 | 具体体现 |
|--------|---------|
| AI 指令系统 | 模板填充 Prompt 可直接集成到现有「指令库」，复用数据结构和 UI |
| Paged.js 渲染管线 | 填充后的内容完全经由现有 A4 渲染，无需新建渲染路径 |
| Word 导出 | 模板输出与普通 Markdown 导出 Word 共享同一套 `exportDocx.ts` |
| `demo version + dirty` 缓存机制 | 模板选择可集成到 `DocumentSettings` 持久化，自动保存用户最后使用的模板 |

### 4.4 优先级建议

**建议优先级：低-中（排在 Word 基础导出之后）**

```
当前优先级队列（建议顺序）：
1. [高] Word 导出基础功能 (Task 1-8)      ← 模板功能的必要前提
2. [中] 扩展文档排版样式（路线图 §2 任务8）
3. [中] 升级 Paged.js（路线图 §2 任务8.5）
4. [低-中] Word 模板填充功能（本报告方向）
5. [低] 多语言适配（路线图 §3）
```

**启动模板功能的门控条件：**
- Word 基础导出（Task 1-8）全部完成并稳定
- 至少有 4 套内置模板骨架编写完毕
- 不依赖任何新的 npm 包（复用现有 `docx` 库）

## 五、推荐实现方向与路线图

### 5.1 MVP 范围（Phase 1）

**目标**：以最小代价验证用户对「模板填充」场景的需求

| 功能项 | 实现方式 |
|--------|---------|
| 内置 4 套模板 | JSON + Markdown 骨架（劳动合同/工作报告/通知函/项目申请书） |
| 模板选择 | 工具栏新增「模板」下拉菜单，简单列表，无缩略图 |

| AI 指令 | 一键生成填充 Prompt，复制到外部 AI 使用 |
| 导出 | 依赖已有 Word 导出功能 |

**MVP 工作量估算**：2-3 天（不含 Word 基础导出的前置工作）

### 5.2 完整功能（Phase 2）

| 功能项 | 实现方式 |
|--------|---------|
| 可视化字段表单 | 基于 TemplateField 定义自动生成表单 UI |
| 占位符高亮预览 | 未填字段在 Paged.js 预览中橙色高亮 |
| 模板缩略图 | 静态 SVG 骨架（与风格指令缩略图技术相同） |
| 扩展至 8-12 套模板 | 覆盖合同/报告/简历/通知/协议/申请书等场景 |
| AI 粘贴结果智能提取 | 自动剥除代码块包裹和多余解释文字 |

### 5.3 明确不做（边界）

- **用户上传自定义 .docx 模板**：解析 .docx 的复杂度过高，且与项目「零后端、自研数据模型」的定位不符
- **在线实时 AI 填充**：项目不集成本地 CLI 或后端 AI，遵循离线工作流约定
- **模板的版本管理/协作编辑**：超出纯前端工具的定位范围
- **从 .docx 模板文件反向解析字段**：技术风险高，工程收益不匹配

## 六、可行性结论

**结论：可行，推荐采用自研模板体系（JSON Schema + Markdown 骨架）方案。**

核心判断依据：

1. **路径最短**：模板填充本质是「带占位符的 Markdown 文档」，与项目现有数据模型（DocumentBlock + Markdown 源）天然契合，不需要引入新的解析层
2. **风险可控**：唯一的外部依赖风险（.docx 导出）已在原调研中完成评估，功能本身（占位符替换 + 表单 UI）零风险
3. **增量迭代**：MVP 可以在不改变任何现有代码的前提下，仅通过「模板数据 + 简单 UI 入口」上线，再逐步完善可视化表单和 AI 指令
4. **与项目定位契合**：「离线 AI 工作流」的模板填充 Prompt 是对现有指令系统的自然延伸，不改变产品形态

### A.2 技术路径修正

原评估提出的 `{{field_id}}` 占位符 + 独立 `renderTemplate()` 正则替换方案，修正为：

**直接复用项目现有的自定义标签解析体系。** 项目已有成熟的标签识别机制（`documentModel.ts` 的 `classifyBlock()` 已识别 `<page-break/>`、`<p-title>`、`<component>` 等自定义标签），模板字段以同类的自定义标识嵌入 Markdown，由现有解析管线统一处理，不引入新的替换层。

**具体含义**：
- 模板字段以项目惯用的自定义标签形式存在于 Markdown 中（而非引入 `{{}}` Handlebars 语法）
- 解析逻辑并入现有 `splitMarkdownBlocks()` / `classifyBlock()` 体系
- 渲染时由现有 Paged.js 管线统一处理，无需新增 `renderTemplate()` 中间层

### A.3 指令库设计（对标自由画布）

A4 文档模式的指令库将扩展为多条指令，每条对应一种文档风格：

```
A4 文档指令库
├─ 基础排版指令（现有，buildDocumentAiGuide）
├─ [风格A] 指令 + 预览    （如：商务报告风格）
├─ [风格B] 指令 + 预览    （如：学术论文风格）
├─ [风格C] 指令 + 预览    （如：政府公文风格）
└─ ...
```

每条指令在指令库中以卡片形式展示，带缩略图预览（复用 `StyleThumbnail` 技术方案），用户点击复制对应 Prompt。

### A.4 封面页指令变体

**「含独立封面页」作为独立指令变体**，与「不含封面页」版本并列存在于指令库中。

含封面页版本的指令需额外要求 AI 生成以下元数据字段：

| 字段 | 说明 |
|------|------|
| 文档版本 | 如 V1.0、V2.3 |
| 编写者 | 作者姓名/部门 |
| 编写时间 | YYYY-MM-DD |
| 审核者 | 审核人姓名/部门 |
| 审核时间 | YYYY-MM-DD |
| 文档状态 | 草稿/评审中/已发布/已归档 |
| 机密等级 | 公开/内部/机密/绝密 |

这些字段在封面页中以结构化方式排版（项目已有封面页识别能力，见 `documentModel.ts` 的 `isCoverPageBlocks()`），指令中需明确告知 AI 封面页的标签结构和字段位置。
