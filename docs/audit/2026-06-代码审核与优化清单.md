## 二、确认要修复的问题（本次执行）

### 1. 【Bug·中】【必做】SettingsModal 表单状态不随打开动作重置

**位置**：`src/components/editor/SettingsModal.tsx:16-27`

**问题**：`form` 用 `useState` 在首次渲染时从 `imageHostConfig` 初始化，组件**常驻挂载**（`if (!isOpen) return null` 在所有 hooks 之后）。当用户打开弹窗→修改了某字段→点「取消」关闭→再次打开时，`form` 仍是上次未保存的脏值，而非 store 中的真实值。用户会误以为配置已保存，或看到与实际不符的表单。

**收益**：消除「取消后重开看到脏数据」的真实困惑，纯前端配置类弹窗的标准做法。

**处置**：在 `isOpen` 由 `false→true` 时重置 `form` 为当前 store 值。

---

### 2. 【资源泄漏·中】【必做】`localImageUrls` 的 Object URL 永不释放

**位置**：`src/lib/editor/imageStorage.ts:111-127`（`resolveImageUrl`）

**问题**：`localImageUrls[id] = URL.createObjectURL(blob)` 创建的 Object URL 会一直常驻内存，删除图片或长时间使用后没有 `URL.revokeObjectURL` 回收路径，造成 Blob 数据在内存中累积泄漏。纯前端工具频繁粘贴/上传图片时尤其明显。

**收益**：避免长时间使用后内存持续增长。提供 `revokeImageUrl` 并在合理时机调用。

**处置**：新增 `revokeImageUrl(id)` 工具函数并在 `removeLocalImage`（如有）/切换图床等场景调用；当前先补上回收能力与导出，保持最小改动。

---

### 3. 【健壮性·中】【必做】`eslint.config.js` 存在但依赖缺失，且配置含不可用规则

**位置**：`eslint.config.js`、`package.json`

**问题（两部分）**：
1. 仓库根目录有一份完整的 flat config，但 `package.json` 既没有 `lint` script，也没有任何 eslint 相关 `devDependencies`。配置形同虚设，CI 与本地都无法执行 lint。
2. **配置本身有 bug**：引用了 `react-hooks/refs`、`react-hooks/set-state-in-effect` 两个规则，但这两条是 `eslint-plugin-react-hooks@7` 才有的新规则；配置其他写法（`reactHooks.configs.recommended.rules`）对应的是 **v5** API。版本与规则名不自洽，导致即使装好依赖也会报 `Could not find "refs" in plugin "react-Hooks"` 直接崩溃。

**收益**：让代码质量门禁从「无法运行」变为「可运行、可发现真实问题」（修复后输出约 1500 条 warn，供后续渐进整改）。

**处置**：
- `package.json` 补 `devDependencies`（eslint@9、typescript-eslint@8、@eslint/js@9、三个插件、prettier-config）与 `lint`/`lint:fix` script，对齐项目使用的 pnpm。
- `eslint.config.js` 移除 v5 不存在的 `react-hooks/refs`、`react-hooks/set-state-in-effect` 规则，保留 `exhaustive-deps` warn 与其余配置。
- 规则统一为 `warn` 而非 `error`，与项目「渐进整改、不阻塞」的现状一致（约 1500 条历史 warn 留待后续）。

---

### 4. 【健壮性·中】【必做】`ImageHostConfig` 类型重复定义

**位置**：`src/lib/store.ts:59-66` 与 `src/lib/editor/imageStorage.ts:286-291`

**问题**：同一份 `ImageHostConfig` 接口在两个文件各定义一次，结构相同但互不引用。未来增删字段极易出现两边不一致的静默 bug。

**收益**：单一数据源，消除维护隐患。

**处置**：`imageStorage.ts` 改为从 `@/lib/store` 导入（store 为领域状态归属方）。

## 三、建议优化项（本次一并执行，均为低风险正收益）

### 5. 【健壮性·低】`clipboard.ts` 的缓存机制有冗余的"伪观察"注释

**位置**：`src/lib/clipboard.ts:33-52`

**问题**：`markUrlCacheDirty` 被导出但**全项目无人调用**（见下文验证）。`urlToIdCacheDirty` 实际上在每次 `getUrlToIdMap()` 调用时都会被读取，但由于 `localImageUrls` 是模块级可变对象，缓存的"惰性重建"本身是对的，只是那段「监听 localImageUrls 变化」的注释与 `markUrlCacheDirty` 导出属于**死代码**，会误导后续维护者以为存在观察机制。

**收益**：移除死代码与误导性注释，降低维护者认知负担。

**处置**：删除未被调用的 `markUrlCacheDirty` 与其误导性注释；保留惰性重建缓存本身。

---

### 6. 【性能·低】`useBlockHeights` 三个 `setState` 可合并为单次更新

**位置**：`src/lib/useBlockHeights.ts:73-112`

**问题**：`measure()` 内连续三次 `setState`（actualHeights / tableRowHeights / tableOverheads），每次都触发一次潜在的 re-render 队列。虽然 React 会批处理，但三次分别的「变化检测」逻辑分散且重复，可读性差。

**收益**：减少状态切片数量，集中变化检测，逻辑更清晰。**风险低**（语义不变）。

**处置**：合并为单一 `measured` 对象 state + 统一变化检测。

> ⚠️ 该 hook 被 CardMode 强依赖（分页计算），改动需保证返回值结构不变，且测试覆盖。结论：**改动收益有限且触及热路径，本次保持原样，仅在文档记录**，避免「为形式而形式」。

---

### 7. 【可维护性·低】`promptLibrary` 的 `handleCopyCustom` 使用 `any` 类型

**位置**：`src/modes/html/PromptLibrary.tsx:117`

**问题**：`const handleCopyCustom = async (inst: any)` 使用了 `any`，而 `customInstructions` store 已有 `CustomInstruction` 类型。

**收益**：类型安全，符合 eslint `no-explicit-any: warn` 的既定约束。

**处置**：替换为 `CustomInstruction` 类型导入。

---

### 8. 【可维护性·低】`uploadToCos` 使用 `any` 回调签名

**位置**：`src/lib/editor/imageStorage.ts:212-213`

**问题**：`(err: any, data: any)` —— COS SDK 自带类型，可改为 `cos-js-sdk-v5` 提供的类型。

**收益**：类型安全；但 COS SDK 类型较复杂，改动收益边际。

**结论**：**本次保持原样**，避免引入 SDK 类型细节导致脆弱。仅文档记录。
