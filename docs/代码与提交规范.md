# 代码与提交流程规范

为保持 `markdown2view` 项目代码的高质量与可维护性，所有协作者（包含人类开发者与 AI 代理）均需遵循以下代码与提交规范。

## 1. 核心开发原则

### 1.1 简单优先 (Simplicity First)
- **拒绝过度设计**：不为“可能发生”的未来场景编写抽象的底层机制，仅为当前被明确需要的功能编写代码。
- **直观的代码结构**：优先采用结构清晰、直接的实现。即便有时会多写几行，也不为了减少行数而写出极其难以理解的“魔法代码”。
- **手术刀式修改**：修复 Bug 或增加新特性时，精确地触碰需要的行，坚决杜绝顺手“重构”未经测试或没有坏掉的相邻代码。

### 1.2 架构与样式原则
- **React 渲染模型**：遵循 React 最佳实践。重度渲染组件必须使用 `useMemo` 或 `useCallback` 避免无关刷新（如 CodeMirror 的配置）。
- **样式隔离与复用**：
  - 动态计算的尺寸、颜色使用内联 `style={{}}` 属于合理行为，不强行迁移。
  - 静态的基础布局与颜色应当复用 Tailwind 类名或抽取至 `index.css`。
- **构建体积保护**：大型库（如 OSS / COS SDK 等云存储库、或是非核心渲染功能相关的巨型插件）**绝不允许在顶部静态 Import**，必须使用 `await import()` 在触发功能时按需加载。

## 2. 代码组织与格式

### 2.1 命名规范
- **组件文件**：使用 PascalCase（如 `PreviewToolbar.tsx`）。
- **工具函数与钩子**：使用 camelCase（如 `useDebounce.ts`, `imageStorage.ts`）。
- **样式类名**：优先使用 Tailwind。自定义类名使用带语境的连字符格式（如 `document-block`, `app-logo-bg`）。

### 2.2 注释规范
- **中文优先**：除了代码中的英文变量外，用于解释业务逻辑、避坑指南、算法原因的注释一律使用中文。
- **为什么，而不是是什么**：注释只写“为什么这么写”（Why），而不是“这行代码在干什么”（What），代码本身应该自我解释它在干什么。

## 3. Git 提交规范 (Commit Convention)

本项目采用标准的 Angular 规范，所有的 Commit Message 应当遵循以下格式：

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
```

### 3.1 Type (类型)
- `feat`: 新增功能 (feature)
- `fix`: 修复 bug
- `docs`: 文档变更
- `style`: 代码格式（不影响代码运行的变动，如空格、分号等）
- `refactor`: 重构（即不是新增功能，也不是修改 bug 的代码变动）
- `perf`: 性能优化
- `test`: 增加测试或修改测试
- `build`: 构建系统或外部依赖变更（如 Vite、npm 包更新、Chunk 分块调整）
- `chore`: 对构建过程或辅助工具和库的更改

### 3.2 示例
```text
feat(article): 增加长图文模式的一键去重功能

为工具栏增加了去重按钮，使用基于正则匹配的方式过滤重复段落。
```

```text
build: 优化 Vite 打包的 manualChunks 策略

分离了 react-vendor 与 codemirror-vendor，解决了单包超 1.1MB 导致的首屏缓存利用率极差问题。
```

## 4. 验收与合并前检查单

每次提交 PR 或合并入主干前，需执行并确保以下命令通过：
1. `pnpm typecheck`：通过 TypeScript 静态类型检查。
2. `pnpm test`：通过全部的 Vitest 单元测试。
3. `pnpm build`：确保构建不报错，并观察终端输出的打包产物体积，确保核心 chunk 没有异常膨胀。
