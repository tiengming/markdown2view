# 开发经验与偏好记录

本文档用于记录项目开发过程中积累的技术经验、踩坑记录以及用户的特定偏好，供后续开发和维护参考。

## 1. 用户偏好与设计准则

- **中文优先**：规划文档、需求说明、流程说明、复盘说明等优先使用中文；代码中的必要注释使用中文。项目暂未支持多语言国际化（i18n）。
- **外链 CDN 资源原则**：AI 生成 HTML 时，若源文本为中文，则生成的 HTML 中**不得引用海外 CDN**（如 unpkg、jsdelivr、Google Fonts 等），须使用国内可访问的资源或系统内置字体栈；若源文本为英文，可酌情考虑但非必要不推荐海外 CDN。
- **简单优先 (Simplicity First)**：用解决问题的最小输出，拒绝猜测性的发散。不做过度抽象。
- **目标驱动执行**：任务执行前定义成功标准，循环迭代直到验证通过。手术刀式修改，只触碰必须改动的部分。
- **UI 审美偏好**：
  - 注重“呼吸感”，避免排版局促。例如分页图文卡片的四周留白应保持全局比例协调统一（如顶部 32px，左右 30px，底部相应留白）。
  - 各模式工具栏注重模块化，不强行把所有按钮堆叠在顶部全局导航，而是根据当前模式的上下文提供最相关的操作（如 AI 指令复制、使用指南整合到局部工具栏）。
  - 界面引导：对于如"自由画布"这种高阶无代码沙箱，需提供直观的操作步骤引导（挑选风格 -> 给大模型 -> 粘回来变现）。
- **设计系统 single source of truth**：UI 视觉决策以项目根目录 `DESIGN.md` 为准（中性色 slate / 强调色 `var(--accent)` / 图标线性 SVG / 禁止 Emoji 作 UI 图标等）。做任何 UI 工作前先读 `DESIGN.md`，复用其中 token 与组件模式，不得擅自发明新的颜色/间距/圆角/字体值。
- **自由画布风格库治理**：
  - 风格库按“输出类型（幻灯片/长页/卡片/报告/仪表盘/文档）→ 视觉气质（极简/编辑/科技/数据/温暖/代码）”组织，避免品牌名和交付物类型混排。
  - 新增风格前先检查是否能归入既有 `family`；除非能明显覆盖新场景，否则不要新增同质化品牌皮肤。
  - 避免“霓虹、毛玻璃、夸张渐变、Emoji、极致动效”等容易导致土味或导出不稳定的表达。
  - 保留分页图文的低门槛路径；自由画布中的多页图文用于复杂品牌风格和深度定制。

## 2. 核心技术经验与踩坑记录

### 2.1 公众号渲染引擎（engine）经验

- **换行处理**：换行必须用 `<br>`，且不能落在 `<span leaf="">` 内部。公众号把带 `leaf` 属性的 span 当作原子文本节点，丢弃内部的 `<br>`。因此 `leaf()` 处理时已按 `\n` 拆成多个 leaf span，`<br>` 作为兄弟节点。不要二次包裹 `leaf()`。
- **顶格输出**：`leaf()` 会对每行 `trim()`，换行后顶格输出。`inlineFormat()` 末尾用正则让 callout/breaking 等正文支持顶格换行。
- **多行标签解析**：`<title>/<p-title>/<lead>` 等支持跨行书写，闭合检测必须**非锚定**（不限于行首），否则标签无限延长。
- **中英文排版**：自动空格用 `pangu()`，已接入 `leaf()` 与 `inlineFormat()`，保证排版规范。

### 2.2 隐藏 DOM 分页与测算（社交卡片模式）

- **DOM 高度测量陷阱（Margin Collapse）**：在卡片高度精确计算切页时，若直接使用 `offsetHeight` 累加每个块，会丢失块与块之间折叠的外边距（Margin）。**必须使用 `offsetTop + offsetHeight - lastBottom`** 来计算包含边距的实际物理高度，避免累计误差导致卡片底端发生溢出遮挡。
- **列表项 (li) 的分页打断**：连续的列表项默认在 Markdown 渲染中属于同一个区块。如果列表过长超过一页卡片，会在分页算法中导致整块卡片下移或直接溢出截断。解决方案是在传入 `parseMarkdown` 之前，将列表项强制替换为段落断开（例如将 `\n- ` 替换为 `\n\n- `），使其成为独立的 Block，从而允许在列表项之间进行卡片切分。
- **卡片物理分页拦截与内容过滤**：在卡片解析引擎中，当遇到物理分页标识 `<page-break/>` 时，必须将其解析为独立的 `pagebreak` 类型块。测算高度时将其强制归零（高度为 `0`），而在 `paginateBlocks` 分页打包后，**必须将该块从内容流中过滤清除**，实现零占位无痕强行分页，保障导出和预览时的渲染干净无杂质。

### 2.2.1 基于 Paged.js 的物理分页与 A4 打印踩坑记录

- **宿主样式隔离与克隆**：
  - A4 文档模式借助 iframe 来完美隔离 Paged.js 的 Polisher 在主页 head 中注入的大量全局 `@page` 样式（否则会破坏主站 UI）。
  - 用 `collectParentStyles()` 获取宿主文档的所有 `<style>` 和 `<link>` 注入 iframe，以实现开发态热更新和生产态的排版同步。
- **表格跨页（thead 重复与续表标记）Handler**：
  - W3C 标准下的表格跨页是由浏览器分页引擎自动完成的。如果我们要往续表的头部加上 `（续表）` 以及重复 `thead` 表头，需要继承 Paged.js 的 `Paged.Handler` 类。
  - **继承机制避坑**：`Paged.Handler` 在 Paged.js 内是个 ES6 class，在 iframe 引导脚本中**必须使用 class extends 继承**，不能使用 ES5 的 `apply()` 伪继承，否则在某些浏览器上会报错 `Class constructor Handler cannot be invoked without 'new'`。
  - **续表逻辑**：在 `afterParsed` 时遍历所有表格并按 `data-ref` 缓存其 thead 结构，在 `afterPageLayout` 中找到带有 `data-split-from` 标记的跨页续表，清理其可能残留的无效 thead，把缓存的 `thead` 插入表头，并动态在表格前面插入一个 `div.continued-caption` 提示“（续表）”文字。
- **重排防抖与高度适配**：
  - 渲染完成后，必须在下一帧重新计算 iframe 的 `getBoundingClientRect().height` 并赋值给其 `style.height`，使得 iframe 本身不带有内部滚动条，而由父级外部容器统一发生滚动。
  - **高度测量坑**：不能使用 `doc.body.scrollHeight`。因为页面开启了 CSS `zoom` 缩放，Chrome 下的 `scrollHeight` 对于带有 `zoom` 缩放子元素的容器会返回其**未缩放前的物理高度**，导致测量高度远大于实际高度，最终使得外层页面底部留出大片无效的巨幅白屏。必须使用 `getBoundingClientRect().height` 获取宿主缩放后的精确真实高度。
- **A4 物理分页实现**：
  - 采用工具栏插入的物理分页符 `<page-break/>` 作为强制分页标志。
  - 在渲染引擎转译 HTML 时，对于 `<page-break/>` 标签，统一将其转译为 `<div class="page-break" style="page-break-after: always; break-after: page;"></div>` 块。Paged.js 在流式分切时，会自动识别该 CSS 指令并在该 div 处强行起新页，从而无缝衔接用户对于强制切页的排版意图。

### 2.3 编辑器（CodeMirror / @uiw）经验

- **选区背景遮挡**：`.cm-activeLine` 必须使用带透明度的颜色（如 `rgba(...)`）。CodeMirror 的选区层 `.cm-selectionLayer` `z-index` 为 -1，如果行背景不透明，会导致文字拖拽选区高亮被遮挡（选得中但看不见）。
- **组件重渲染卡顿**：CodeMirror 的 `extensions` 属性必须用 `useMemo` 缓存。如果在每次 render 时生成新数组，会触发反复 `reconfigure`，造成输入严重卡顿。
- **高度填充**：使用 `height="100%"` 时，插件会让 `.cm-scroller` 采取 `height:100%!important` 以正常充满外部弹性面板。
- **防抖回写把外部更新回滚（"恢复示例无效"根因）**：各模式采用「本地 state + `useDebounce` 防抖回写 store」的双向同步。回写 effect 若把 store值（`markdown`/`html`）放进依赖，并与**滞后的** `debounced` 值比较，则当 store 被外部更新（恢复示例 / 版本同步）时，effect 会立即触发并用旧的 `debounced` 值把 store 写回旧值，造成外部更新被回滚、表现为"点了没反应"。**正确做法**：用一个 `ref` 持有最新 store 值，回写 effect **仅依赖 `debounced` 与 setter**，比较时读 `ref.current`，从而只在真正的本地编辑发生时回写。

### 2.4 HTML 沙箱与自适应拉伸经验

- **Zoom 与 Transform 的选型与坑**：
  - 在缩放整个页面内容以适应屏幕时，对于单页内容（长文本），使用 CSS `zoom` 缩放更好，因为它能实际改变布局尺寸，从而正确触发浏览器的原生滚动条。
  - 对于多页模式（类似幻灯片，元素含有 `.page` 或 `.slide`，且外层 Flex 居中），**千万不要对外层 `body` 使用 `zoom` 配合 `min-height: 100vh`**。在 Chrome 下，`100vh` 永远基于物理视口计算，不随 `zoom` 缩放，导致 `body` 在 `zoom: 2` 时变成物理视口的两倍高，使居中的幻灯片严重偏下并产生滚动条。
  - **解决方案**：针对多页模式，通过 JS 注入 `--auto-scale` 变量，使用 `transform: scale(var(--auto-scale))` 仅对幻灯片本体进行居中缩放，完美解决 100vh 错位和溢出截断问题。
- **不可见 DOM 的尺寸测量**：当使用 `display: none` 隐藏非当前页以实现翻页时，隐藏节点的 `offsetWidth` 和 `offsetHeight` 会变成 `0`。在 `ResizeObserver` 中测量尺寸时，必须通过 `find(el => el.style.display !== 'none')` 来抓取当前可见页的尺寸，否则会导致缩放逻辑全部失效。
- **导出画质保护**：`modern-screenshot` 的 `domToJpeg` 会直接截取带有 `zoom` 或 `transform` 的最终渲染效果。如果要导出高清 1x 设计稿，必须在执行导出函数的瞬间，将 `zoom` 和 `--auto-scale` 临时重置为 `1`，在 `finally` 块中再恢复。
- **模板字符串编译陷阱**：在使用工具对代码进行大段替换时，Vite/React 的 Babel 编译对含有错误转义符的字符串极为敏感（如双重转义的 `\${scale}` 报错为 Unicode escape 错误），容易引发难以排查的语法树崩溃。推荐在操作动态变量时，优先使用数字运算和 `.toString()` 拼接，避免在模板字符串里制造转义灾难。

### 2.5 A4 题注与排版解析优化经验

- **题注误判防范与上下文校验**：为防范 AI 在正常段落的换行中，将描述性长句子（如“图 1 展现了……”）误判为题注而导致段落截断与错误居中，我们建立了基于上下文、前缀格式以及关键词的复合校验机制：
  - **表格题注**（表/Table）必须在**下方 3 行内**存在 Markdown 表格首行（含有 `|` 且次行含有分隔符）。
  - **图片题注**（图/Fig/Figure）必须在**上方 3 行内**存在 Markdown 图片（`![`）或 HTML 图片（`<img`）。
- **Markdown 表格空单元格错位与消失（`filter(Boolean)` 的滥用陷阱）**：在实现底层的 Markdown 到 HTML 解析引擎时，经常会在把行内容按 `|` 切分成数组后，习惯性地加上 `.filter(Boolean)` 来清除首尾残余的空字符串。然而，对于本身包含真实空单元格的表头（如 `| | 不含税 | 含税 |`），`.filter(Boolean)` 会直接将其“清洗”掉，导致解析出来的表头列数少于数据行数。当表头 `<th>` 数量少于 `<td>` 时，浏览器会自动采用错误的首列覆盖渲染，使得整个表格的样式表现严重错乱（左侧第一列边框和头部内容全部消失）。**解决方案**：永远不要对表格数组进行全局 `.filter(Boolean)` 过滤，必须采用“定点剥离”（检测开头结尾的 `|` 并去除），再通过 `split('|')` 划分并 `trim()` 保留原始的空字符串，严格保持单元格数量的一致性。
  - **空格分隔题注限制**：如果分隔符仅为普通空格（无冒号或点，如 `图 1 结构图`），则长度限制在 60 字符内，且内容不得包含 `展示|显示|展现|是|有|如下图|如下表` 等描述性动词，否则判定为普通段落。
  - **排版包裹字符清洗**：AI 书写题注时常带有加粗或斜体（如 `**图 1: xxxx**`），在进行题注正则匹配前，需剥除两端的排版字符进行校验，校验通过后再对包含排版字符的整行做富文本格式化，以达到“居中 + 加粗/斜体”的完美版式。

### 2.6 示例内容缓存与版本同步（demo cache）

- **缓存来源**：`zustand persist` 把整个 store（含 `articleMarkdown`/`documentMarkdown`/`cardMarkdown`/`html`）写入 `localStorage['m2v-store']`。因此更新 `src/data/demo*` 源码后，**已有缓存的老浏览器不会自动看到新示例**——这不是 HTTP 缓存。
- **版本号 + dirty 标记机制**（见 `src/lib/store.ts`）：
  - `DEMO_VERSION`：更新示例并希望老用户下次加载自动获取最新示例时，将其 +1。
  - 每个内容字段配 `*Dirty` 标记：用户在编辑器输入时（经 setter）置 `true`；`restoreDemo` / 注入示例时置 `false`。
  - 挂载时 `syncDemoContent(DEMOS)`：仅当 `demoVersion !== DEMO_VERSION` 时，刷新**未被用户编辑过**（dirty=false）的字段为最新示例，并写入新版本号；已编辑内容始终保留。
  - 「恢复示例」走 `restoreDemo(mode, DEMOS)`，强制写入当前模式最新示例并清 dirty。
- **一次性迁移说明**：对此机制上线前已存在 `m2v-store` 的浏览器，因历史上未跟踪 dirty，会被视为未编辑而在首次加载时统一刷新为最新示例（一次性覆盖）。这是为满足"老浏览器也能拿到新示例"的取舍，之后 dirty 跟踪即精确。

### 2.7 Lint 告警治理：CSS 兼容性与内联样式

- **`text-justify` 兼容性告警**：`text-justify: inter-ideograph` 仅旧版 IE / 老 Edge 支持，Chrome / 现代 Edge / Safari / Opera 均不支持，会触发 linter 兼容性告警。对 CJK 文本而言，**仅靠 `text-align: justify` 即可实现两端对齐**，因此直接删除该属性即可消除告警，视觉无回归（见 `src/index.css` 的 `.document-block p`）。
- **「CSS inline styles should not be used」告警的处理原则**：该规则会对所有 `style={{}}` 报警。处理时按「静态 / 动态」分类，避免过度工程：
  - **静态样式**（值固定、不依赖运行时状态）应抽到 `index.css` 的类中。例如：`App.tsx` 的 logo 背景 / 标题强调色复用既有 `.app-logo-bg`、`.app-title-accent`；`PromptLibrary` 复制按钮用新增 `.accent-bg`；`ArticlePreview` 手机框抽到 `.phone-frame`。
  - **动态样式**（值由运行时计算，如 `DocumentMode` 的页宽/页高/页边距、`CardMode` 的 `size.w/size.h`、主题色板的 `t.accent`、风格卡片的 `s.accent`）属于 React 惯用的内联样式，**强行迁移到 CSS 反而损害可读性、并不会消除告警**（只要保留 `style` 属性 linter 仍会报），故保留内联。
  - **结论**：内联样式告警不应一刀切清零；只迁移真正静态的部分，动态值保持内联是正确的工程取舍。

### 2.8 交互优化与组件架构经验

- **工具栏统一提取**：不同模式（如长图文、卡片、自由画布等）的右侧功能入口应通过抽象的公共组件（如 `PreviewToolbar.tsx`）进行集中管理。约束操作按钮接口结构，统一 Tooltip 弹出方向和按钮视觉间距，从而消除早期硬编码带来的体验不一致，并大幅降低后续维护的冗余度。
- **界面引导与视觉聚焦**：对于提供效果预览的交互组件（如风格预览模态框），应尽最大可能隐藏复杂的底层参数源码，让预览页面本身占据 C 位。搭配显眼的“📋 复制并去 AI 生成”大按钮，完成操作闭欢。让专业配置对普通用户隐匿，是保持工具“极简高级感”的关键。
- **自定义指令全局化与兼容**：Zustand 的本地化指令不仅应支持单一模块，且应该在数据结构设计初期就带上 `mode` 字段以便隔离不同场景的高质量提示词。同时注意兼顾老版本的历史数据平滑迁移（如将无 mode 标记的指令默认归类为 html 模式）。

### 2.9 构建产物体积优化 (Chunk Splitting) 经验

- **防范默认打包的超大 Chunk**：对于集成 CodeMirror 核心、语法树及 Highlight.js 的编辑器类应用，Vite 的默认打包极易产生大于 1MB 的业务混合 Chunk，导致微小业务变更破坏了整个大库缓存。
- **手动分包机制**：在 `vite.config.ts` 的 `manualChunks` 中精准切分 `react-vendor`、`codemirror-vendor`、`engine-vendor`。既消除了超 500KB 报警，也极大提高了大体积依赖的浏览器端二次加载长效缓存利用率。
- **SDK 按需加载原则**：涉及第三方云存储（如阿里云 OSS ~690KB、腾讯云 COS ~168KB）这类体积庞大且仅在特定动作下才触发的包，**坚决不要在顶部静态导入**，必须采用 `await import('ali-oss')` 的方式在使用时动态加载，保障极速首屏体验。

### 2.10 风格指令缩略图与交互简化经验

- **缩放比例选型与视觉可读性平衡**：缩略图为了承载丰富的内容版式信息，通常采用缩小整体容器（如 `scale` 缩放）的微型网页预览实现。如果缩放过大（如 `scale(0.5)`），由于视口高度急剧压缩，所有原本在小容器中设计的文字、图形将进一步缩小，导致细节丢失和视觉“乱码”。**解决方案**：采用轻度缩放 `scale(0.8)` 配合放大版面初始字号（大标题设计为 `15px ~ 18px`，细节与标注限制在 `11px ~ 12px`），既保留了网页在卡片中的微型版式精致感，又确保了核心标题与视觉元素的直观可读性。
- **纯 CSS 卡片自适应平铺**：卡片在前端多处于网格布局（Grid）中，其宽度随着屏幕尺寸动态响应式拉伸。如果内层缩略图容器使用固定的像素宽度（如 `320px`）搭配固定比例缩放，当外层网格卡片比其缩放后的尺寸宽时，卡片右下侧将产生巨大的刺眼白边。**解决方案**：利用纯 CSS 比例推算，将内层绝对定位容器的宽度和高度设置为外层物理边界的 `125%`（与缩放比例 `0.8` 的倒数一致：`1 / 0.8 = 1.25`），再通过 `transform-origin: top left` 和 `scale(0.8)`，可使用纯 CSS 实现**任何物理宽度与高宽比下的完美响应式自适应铺满**，且无需任何 JS resize 监听运行时。
- **交互减法原则（拒绝不完美的预览弹窗）**：对于类似海报长图、小红书竖版卡片等多场景排版，使用通用尺寸（如 16:9 或 4:3 窗口）去提供“全屏预览示例”时，常会因为弹窗宽高与内容版式比例不符，导致版式拉长错位、内容不铺满或出现大量上下留白。如果缩略图卡片本身已经被优化得足够直观大气，应**果断移去多余的 Hover 遮罩和预览弹窗**，让卡片直接展示并保留复制按钮，以此降低无意义的 DOM 层级、避免不可控的弹窗错位，完成最简洁的无干扰操作闭环。

### 2.11 多卡片异步并发压缩导出（Web Worker 与 fflate 异步化）

- **fflate 压缩阻塞主线程问题**：在导出多张卡片为 ZIP 包时，若使用 `fflate` 的 `zipSync` 同步压缩接口，当卡片数量较多且图片体积较大时，压缩会严重占用 CPU 计算资源并导致前端主线程冻结，页面产生明显的假死甚至浏览器无响应。
- **Web Worker 异步并发架构**：
  - 将 `zipDownload.ts` 重构为基于 `fflate.zip` 异步多线程执行压缩的方案，从而避开了主线程同步压缩的瓶颈。
  - 使用 `Promise.all` 配合 `ArrayBuffer` 的并发下载和抓取图片 Blob，实现将所有图片数据预加载并转换为 `Uint8Array` 字节流，随后直接传递给 `fflate` 引擎处理。
   - 该方案极大地缩减了多文件导出 ZIP 时的等待时长，并保证了压缩打包的全程 UI 毫无卡顿，提供极佳的非阻塞式导出体验。

### 2.12 自由画布（HTML）模式渲染性能优化经验

- **渲染慢的根本原因分析**：
  - **播放正常但渲染慢**：播放功能直接在 iframe 中执行动画，不涉及截图操作。而"渲染"和"PDF 导出"都依赖 `captureElementInIframeToBlob` 截图函数，这是核心瓶颈。
  - **onLoad 回调延迟**：`HtmlMode.tsx` 的 `onLoad` 回调中曾有 500ms 固定延迟（`setTimeout`），iframe 的 `onLoad` 事件触发时 DOM 已经解析完成，无需额外等待。
  - **重复 DOMParser 解析**：`expectedPageCount` 和 `htmlTitle` 曾各自独立调用 `DOMParser().parseFromString()`，且依赖 `[html, localHtml]` 导致每次按键都执行。应合并为一次调用，并使用 `debouncedHtml` 作为依赖。
  - **缩放逻辑定时器过多**：`stabilizeScale` 函数中曾有 5 个定时器 `[0, 50, 150, 350, 800]`，每次内容变化触发 5 次 DOM 测量和样式修改。已优化为 `[0, 100, 400]`（3 个）。

- **截图等待链优化**（`exportImage.ts`）：
  - `waitForDocumentReady` 中的资源等待（样式表、字体、图片）从**串行**改为**并行**（`Promise.all`）。
  - 各项超时时间统一从 6-8 秒降低到 3 秒：readyState、样式表、图片、waitUntilLoad。
  - `waitForStability` 的 MutationObserver 稳定检测时间从 150ms 降到 100ms，等待帧数从 3 降到 2。
  - `captureElementInIframeToBlob` 中的 DOM 稳定等待从 1000ms 降到 500ms。

- **防抖延迟**：`useEditorDocSync` 的防抖延迟为 500ms，这是设计值，保证输入流畅。渲染使用 `debouncedHtml` 而非 `localHtml`，避免每次按键都触发 iframe 重新加载。

- **关键经验**：
  - iframe 的 `srcDoc` 更新会导致整个 iframe 重新加载，这是浏览器行为，无法避免。
  - 使用 `debouncedHtml` 作为 `useMemo` 的依赖，而非 `localHtml`，可将 DOM 解析频率从每次按键降到约 500ms 一次。
  - 缩放逻辑中的定时器应尽量减少，ResizeObserver 回调已足够覆盖大部分场景。

### 2.13 Word 导出（docx）经验

- **技术选型**：采用 `docx`（dolanmiu/docx）库，动态导入避免影响首屏体积。该库支持完整的 OOXML 生成，API 与 DocumentBlock 模型天然契合。
- **字体映射**：中文字体映射到 Word 字体名（SimSun、FangSong、Microsoft YaHei），同时设置 ascii 和 hAnsi 为 Times New Roman，保证中英文混排效果。
- **字号映射**：CSS 像素到 half-point 的换算公式：`Math.round(px × 1.5)`。三档字号（small/normal/large）对应不同的 body 和 h1-h3 字号。
- **公式渲染**：使用 MathJax tex2svg 渲染为矢量 SVG，同时生成 PNG fallback 以兼容旧版 Word。公式预处理将 `$`/`$$` 包裹的 LaTeX 转换为代码样式，再通过公式映射表替换为图片。
- **表格转换**：复用现有 `parseTableMarkdown` 解析 Markdown 表格，设置 `tableLayoutType: FIXED` 防止 Word 自动调整列宽。表头行设置 `tableHeader: true` 以支持跨页重复。
- **图片嵌入**：fetch 图片 URL 获取 ArrayBuffer，通过 `createImageBitmap` 获取原始尺寸，等比缩放至内容区宽度。CORS 失败时降级为占位文本。
- **页眉页脚**：通过 tab stop 实现左右分栏页眉，解析 footerText 模板中的 `{page}` 和 `{total}` 占位符，替换为 `PageNumber.CURRENT` 和 `PageNumber.TOTAL_PAGES`。
- **封面页检测**：检测第一个 `pagebreak` 前的块结构，若仅包含一个标题和若干段落/表格，则判定为封面页，设置 `titlePage: true` 并使用独立的页眉页脚（空白）。
- **分页符处理**：将 `<page-break/>` 转换为 `new PageBreak()`，插入到段落中。

### 2.14 blockParser 提取经验

- **问题背景**：A4 文档、卡片、长图文三个模式下，同一套自定义标签解析逻辑（`splitMarkdownBlocks` + `classifyBlock`）被重复实现，维护成本随模式数量线性增长。
- **三层解析结构**：块拆分（将 Markdown 字符串拆分为语义块）、块分类（识别 block 类型）、渲染（将 block 转换为 HTML）。其中前两层在 A4 文档和卡片模式中几乎逐行重复。
- **抽象方案**：提取 `engine/blockParser/` 公共模块，包含 `splitBlocks.ts`、`classifyBlock.ts`、`types.ts`、`index.ts`。A4 文档和卡片模式改为 import 调用。
- **差异化保留**：A4 文档保留 `mergeCaptionBlocks()`（题注合并）、`estimateBlockHeight()`（像素高度估算）、`paginateDocumentBlocks()`（物理分页）；卡片保留 `estimateBlockUnits()`（内容单位估算）、`paginateBlocks()`（内容单位分页）。
- **统一正则**：`image` 使用更严格的 `/^!\[/`；`pagebreak` 在拆分阶段显式识别（不区分大小写）。
- **列表项拆分统一**：列表项拆分纳入共享模块，卡片模式不再需要预插入空行拆分列表。
- **验证结果**：`documentModel.test.ts` 16 个用例全部通过，`cardModel.test.ts` 4 个用例全部通过，全量 51 个测试用例零回归。
