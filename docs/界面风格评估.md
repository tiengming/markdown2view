# 界面风格评估

> 更新日期：2026-06-16
> 评估对象：markdown2view 应用外壳（顶栏 / 模式切换 / 工具栏 / 预览区 / 弹窗）
> 对照基准：`.agent/skills` 前端设计指引（`web-design-guidelines`、`frontend-dev`、`awesome-design-md`）及项目自身偏好（`docs/开发经验与偏好记录.md`）
> 用途：本文档为**评估与建议**，待你确认后再决定是否实施、实施哪些项。

---

## 0. 评估范围与方法

- **范围**：应用级界面（chrome），不含各模式预览内容（预览区是用户产出物，受排版规范单独约束）。
- **方法**：逐文件通读 `src/components/**` 与 `src/modes/document`，对照 `.agent/skills` 的设计原则与项目已记录的审美偏好，标注「一致 / 偏差 / 可优化」。
- **不在范围**：功能优化（PWA / 字体本地化 / 防抖自适应等，已记录于 `项目优化审查报告.md`，无重叠）。

---

## 1. 整体定性

当前界面风格定位清晰：**克制的工具型产品气质**——白底、slate 中性色、单一主题色强调、信息密度适中。这与 `design-system` 中"focused / structured / gentle / premium but restrained"的目标，以及 Linear / Vercel / Notion 一类开发者工具的审美取向是**一致的**。

骨架是健康的，没有方向性问题。下面的问题集中在**一致性收口**与**细节质感**两类，属于「打磨」而非「重做」。

---

## 2. 发现清单（按优先级）

### 🔴 P0 — 一致性硬伤（与既有规范/偏好冲突）

#### 2.1 工具栏图标用 Emoji，违反设计原则与自身偏好
**证据**
- `DocumentMode.tsx:118` 复制排版指令用 `✨`；`:131` 导出用 `💾`；`:138` 导出 PDF 用 `🖨️`。
- 对照 `frontend-dev` 设计原则：*"Icons: Use Lucide React, never emoji"*。
- 对照自身偏好（`开发经验与偏好记录.md:18`）：*"避免 Emoji 等容易导致土味的表达"*。
- 对照顶栏与编辑器工具栏（`AppHeader.tsx` / `EditorToolbar.tsx`）：这两处已经全部用线性 SVG 图标，唯独预览工具栏的 action 用了 Emoji。

**影响**：三处入口视觉风格与全局不统一；Emoji 在不同操作系统下渲染不一致（Win/Mac/Android 各异），削弱「高级感」。

**建议**：统一替换为与顶栏一致的 14–16px 线性 SVG 图标（复制→clipboard/复制箭头、导出源→download、导出 PDF→printer）。`PreviewToolbar` 的 `ToolbarAction.icon` 当前定义为 `icon?: string`，建议改为 `icon?: ReactNode` 以承载 SVG。

> 这是**自身规范层面**的冲突，建议无论是否做其它优化都先收口。

---

#### 2.2 复选框强调色用蓝色，与主题色变量脱节
**证据**
- `DocumentMode.tsx:192,200` 复选框硬编码 `text-blue-600 focus:ring-blue-500`。
- 全局已有 `--accent` 主题色变量（可在顶栏切换，默认 `#27ae60` 绿色），按钮 / ModeTabs / Input / Select 均已接入。

**影响**：用户切换主题色后，复选框仍是蓝色，是唯一不跟随主题的交互元素。

**建议**：复选框 `accent-color: var(--accent)`（现代浏览器原生支持，比 `text-blue-600` 更简洁），或保持中性 slate-600 但去掉蓝色。

---

### 🟡 P1 — 质感细节（提升「高级感」）

#### 2.3 输入控件缺少 focus-visible 轮廓，仅靠边框变色
**证据**
- `Input.tsx:7` / `Select.tsx:7`：`focus:border-[var(--accent)]`，无 `:focus-visible` 环。
- 对照 `web-design-guidelines` 无障碍项：检查 focus 样式是否可见；`frontend-dev`：focus 状态需可见。

**影响**：键盘 Tab 聚焦时辨识度偏弱（尤其主题色与边框对比度不高时）。

**建议**：补充 `focus-visible:ring-2 focus-visible:ring-[var(--accent)]/30`。代价极小。

---

#### 2.4 主题色板选中外圈用半透明黑色，视觉略脏
**证据**
- `AppHeader.tsx:176-180`：选中态 `borderColor: '#111'` + `outline: '2px solid #1118'`（半透明黑描边）。
- 圆点本身是彩色，外圈叠半透明黑，在浅色主题（如黄）上会发灰。

**建议**：选中态改用一个独立的 2px 实色环（`box-shadow: 0 0 0 2px white, 0 0 0 4px var(--accent)` 双层投影），既清晰又不脏色。

---

#### 2.5 文档预览区背景为实色 `bg-gray-200`，与全局 `slate` 体系混用
**证据**
- `DocumentMode.tsx:212` `bg-gray-200`、`:237` `bg-gray-200`（作为 grid 分隔缝）、`:251` `bg-slate-100`。
- 全局 `body` 为 `#f3f4f6`（gray-100），顶栏/编辑器工具栏走 slate 系。

**影响**：`gray` 与 `slate` 是两套中性色（gray 偏暖、slate 偏冷蓝），同屏并置会有微妙的「不干净」感。

**建议**：统一到 slate 体系（`bg-slate-200`），让「编辑器 | 预览」两栏的灰背景属同一色相。

---

### 🟢 P2 — 锦上添花（可选）

#### 2.6 Button 组件缺 `focus-visible` 与键盘可达性
**证据**：`Button.tsx:10` base 样式只有 `focus:outline-none`，没有可见焦点环。

**建议**：补 `focus-visible:ring-2 focus-visible:ring-offset-1`。低优先级，因多数场景 Button 外层包了 Tooltip。

---

#### 2.7 主标题在窄屏切换 `markdown2view`→`m2v` 为硬切
**证据**：`AppHeader.tsx:72-80` 在 `headerWidth >= 1300` 时显示全称，否则 `m2v`，无过渡。

**建议**：可接受现状（功能性收缩）。若追求丝滑可做 `max-w` + `overflow` 过渡，收益有限，不建议优先。

---

#### 2.8 错误兜底用 `alert()`，与 Toast 体系不一致
**证据**：`EditorToolbar.tsx:47` 图片上传失败用 `alert()`；全局已有 `Toast` 组件。

**建议**：图片上传失败改走 `onToast` 通道（需把 `onToast` 透传进 `EditorToolbar`）。统一反馈渠道。

## 4. 做得好的地方（保持现状，勿动）

- **顶栏信息密度与响应式断点**（960 / 1300 / 1450 三级渐进折叠）处理成熟，小屏自动收进抽屉，逻辑严谨。
- **ModeTabs 分段控件**（slate-100 容器 + 白底选中态 + 主题色文字）是教科书式的 segmented control，符合 Linear/Vercel 风格。
- **骨架屏**（A4 呼吸动画 / 自由画布光泽扫过）区分两种加载态，细节到位，优于多数同类工具。
- **主题色全程用 CSS 变量 `--accent`**，切换主题色即时生效，架构正确。
- **工具栏模块化**（`PreviewToolbar` 抽象 + 局部上下文工具）符合自身偏好记录的「不在顶部堆全部按钮」原则。
