# 自定义标签解析模块抽象评估

> 本文档基于 2026-06-17 对 markdown2view 项目中 A4 文档、多页图文（卡片）、
> 长图文三个模式下自定义标签解析逻辑的审计结论，提出抽象方案与实施建议。

---

## 一、问题背景

项目在多个模式下使用「自定义标识/标签 → 解析/渲染」体系，例如：

- `<page-break/>` — 强制分页
- `<p-title>` — 文档标题组件
- `<component>` — 自定义渲染组件
- 以及 `markdownParser.ts` 中 20+ 种其他自定义标签

**同一套标签解析逻辑被重复实现在多个文件中**，导致维护成本随模式数量线性增长。

---

## 二、现状审计

### 2.1 三层解析结构

三个模式的标签处理分为三层：

| 层 | 职责 | 代码位置 |
|----|------|----------|
| **块拆分** | 将原始 Markdown 字符串拆分为语义块（block） | `splitMarkdownBlocks()` |
| **块分类** | 识别每个 block 的类型（`heading`/`image`/`table`/`pagebreak`/...） | `classifyBlock()` |
| **渲染** | 将分类后的 block 转换为 HTML（含自定义标签的特殊渲染） | `parseMarkdown()` |

### 2.2 重复情况一览

| 层 | A4 文档 (`documentModel.ts`) | 卡片 (`cardModel.ts`) | 长图文 (`markdownParser.ts`) | 是否重复 |
|----|------------------------------|------------------------|--------------------------------|---------|
| 块拆分 | `splitMarkdownBlocks()` | `splitMarkdownBlocks()` | —（走 CodeMirror 解析） | ✅ **几乎逐行重复** |
| 块分类 | `classifyBlock()` | `classify()` | — | ✅ **正则模式一致** |
| 渲染 | — | — | `parseMarkdown()` | ❌ 不重复（长图文独有渲染引擎） |

### 2.3 重复代码细节对比

**`splitMarkdownBlocks()` 核心逻辑（两份几乎一致）：**

```
归一化换行（\r\n → \n）
  → 识别代码围栏（```）状态
  → 识别开标签（<xxx> 未闭合）
  → 识别 <page-break/> 分页标记 → flush
  → 空行 → flush
  → 列表项拆分（document 有，card 无）
  → mergeCaptionBlocks（document 有，card 无）
```

**`classifyBlock()` / `classify()` 正则模式（完全对应）：**

| 类型 | documentModel.ts | cardModel.ts |
|------|-------------------|---------------|
| `pagebreak` | `/^<page-break\s*\/?>/` | 同上 |
| `heading` | `/^#{1,6}\s/` 或 `<title>` 或 `<p-title>` | 同上 |
| `code` | `/^```/` | 同上 |
| `image` | `/^(!\s*!\[|!\[)/` | `/^!\[/` |
| `quote` | `/^>/` | 同上 |
| `list` | `/^([-*+]\s|\d+\.\s)/` | 同上 |
| `table` | `|` + `|---|` 检测 | 同上 |
| `component` | `/^<\w[\s\S]*<\/\w/` | 同上 |

## 三、推荐抽象方案

### 3.1 第一层：提取 `engine/blockParser/`（高收益、低风险）

将 `splitMarkdownBlocks` + `classifyBlock` 提取为独立模块，
`documentModel.ts` 和 `cardModel.ts` 改为 import 调用。

**提取后目录结构：**

```
src/engine/blockParser/
  ├── splitBlocks.ts        ← 统一的 splitMarkdownBlocks
  ├── classifyBlock.ts     ← 统一的 classifyBlock（含 DocumentBlockKind 类型）
  ├── types.ts            ← DocumentBlockKind / DocumentBlock 等共享类型
  └── index.ts            ← 统一导出
```

**`types.ts` 内容（原在 `documentModel.ts` 中定义）：**

```typescript
export type DocumentBlockKind =
  | 'heading'
  | 'paragraph'
  | 'image'
  | 'table'
  | 'code'
  | 'quote'
  | 'list'
  | 'component'
  | 'rule'
  | 'pagebreak'

export interface DocumentBlock {
  id: string
  kind: DocumentBlockKind
  markdown: string
  estimatedHeight: number
  avoidBreak: boolean
}
```

**各模式保留的差异化逻辑（不提取）：**

| 模式 | 保留内容 | 原因 |
|------|---------|------|
| A4 文档 | `mergeCaptionBlocks()` | 仅文档模式需要处理题注合并 |
| A4 文档 | `estimateBlockHeight()` | 像素高度估算（文档模式专属） |
| A4 文档 | `paginateDocumentBlocks()` | 物理分页逻辑 |
| A4 文档 | `splitTableByHeight()` | 表格跨页拆分 |
| 卡片 | `estimateBlockUnits()` | 内容单位估算（卡片模式专属） |
| 卡片 | `paginateBlocks()` | 内容单位分页逻辑 |

### 3.2 第二层：标签注册表（暂不做，等需求驱动）

`markdownParser.ts` 中的 20+ 自定义标签渲染逻辑目前是 if-else 链：

```typescript
// 当前写法（markdownParser.ts 中大量出现）
if (tagName === 'p-title') { /* 渲染标题 */ }
else if (tagName === 'page-break') { /* 渲染分页 */ }
else if (tagName === 'alert') { /* 渲染提示框 */ }
// ... 20+ 个 else if
```

**理论上可改为注册表驱动：**

```typescript
// 理想写法（暂不实施）
const tagRegistry: Record<string, TagRenderer> = {
  'p-title': renderPTitle,
  'page-break': renderPageBreak,
  'alert': renderAlert,
  // ...
}
```

**但暂不建议实施**，原因：
1. 涉及核心渲染引擎，改动面大
2. 当前 if-else 链可读性好，不存在维护痛点
3. **等待实际需求驱动**：模板填充功能如需注册新标签，届时再重构为注册表

## 四、实施计划

### 4.1 第一阶段：提取 blockParser（推荐立即实施）

**前置条件：** 无（不依赖任何其他功能）

**工作量估算：** 0.5 天

**步骤：**

1. 创建 `src/engine/blockParser/` 目录及上述 4 个文件
2. 将 `documentModel.ts` 中的 `splitMarkdownBlocks`、`classifyBlock`、`DocumentBlockKind`、`DocumentBlock` 移入 `blockParser/`
3. 对齐 `cardModel.ts` 中 `splitMarkdownBlocks` 的差异（主要是列表项拆分和 caption 合并，需统一行为）
4. 修改 `documentModel.ts` 和 `cardModel.ts` 的 import 路径
5. 运行现有测试（`documentModel.test.ts` + `cardModel.test.ts`）确保无回归

**注意事项：**
- `cardModel.ts` 的 `classify()` 对 `image` 的正则比 `documentModel.ts` 更严格（`/^!\[/` vs `/^(!\s*!\[|!\[)/`），提取时需统一为更严格的版本
- `documentModel.ts` 有 `mergeCaptionBlocks()` 调用，`cardModel.ts` 没有，提取后 `splitBlocks.ts` 应不支持 caption 合并（由调用方在拆分后自行处理）
- 两份 `splitMarkdownBlocks` 对 `<page-break/>` 的处理方式存在细微差异：card 把它作为独立 block flush 出来，document 在 `classifyBlock` 阶段才识别。提取时需要统一这个行为，用测试保障不回归

### 4.2 第二阶段：标签注册表（待办，不排期）

**触发条件（满足任一即可启动）：**
- 模板填充功能需要注册 ≥3 种新自定义标签
- `markdownParser.ts` 中 if-else 链超过 30 个分支
- 出现动态标签加载需求（用户自定义标签）

**暂不实施。**

## 五、决策结论

| 决策项 | 结论 |
|--------|------|
| 是否抽象块解析逻辑 | ✅ **是**，提取为 `engine/blockParser/` |
| 是否抽象渲染层（if-else 链） | ❌ **暂否**，等实际需求驱动 |
| 实施时机 | **无前置依赖**，可随时启动 |
| 风险等级 | 🟢 低（有完整测试覆盖） |
