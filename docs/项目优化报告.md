## 一、安全与合规

### 1.1 云存储密钥暴露在客户端并被持久化 ⚠️ P0 未完全关闭

**文件**：`src/lib/store.ts:61-66` 与 `src/lib/editor/imageStorage.ts`

**问题描述**：`ImageHostConfig` 直接在前端保存 OSS 的 `accessKeySecret` 与 COS 的 `SecretKey`，且通过 Zustand persist 写入 `localStorage`。只要用户打开 DevTools 或遭遇 XSS，密钥即可被读取，属于生产环境不可接受的风险。

**建议**：上传能力改为后端代理或临时预签名 URL 方案；前端只保留不敏感的配置项（如 bucket、region）。如必须保留前端上传，也应由后端签发限时 STS 临时凭证，并设置最小权限策略。

**落地状态**：❌ 尚未落地

---

### 1.2 AI 返回 HTML 存在 XSS 风险 ⚠️ P0 部分落地

**文件**：`src/lib/extractHtml.ts`、`src/lib/htmlSanitizer.ts`

**问题描述**：`extractHtml` 对 AI 输出几乎不做净化：只要以 `<` 开头就直接返回，并注入 iframe 渲染。恶意脚本、事件处理器或 `javascript:` 链接均可能在预览/导出阶段执行。

**建议**：
- 引入 `DOMPurify` 对 HTML 做净化，默认禁用 `script` 与事件属性。
- 若业务确实需要执行脚本，使用 `iframe[srcdoc]` 配合 `sandbox="allow-scripts"`，并设置严格 CSP。
- `previewHtml` 中的 link/style/script 注入改用 `DOMParser` + 程序化 DOM 操作，避免字符串正则切分带来的边界错误。

**落地状态**：⚠️ 部分落地 - 已引入自研 `htmlSanitizer.ts` 替代完全无净化，但未使用业界成熟的 DOMPurify。自研方案基于 DOMParser，拦截 script、事件处理器、`javascript:` 协议，但长期维护成本和边界覆盖不如 DOMPurify。建议评估是否替换为 DOMPurify，或在自研方案中补充更多边界用例测试。

---

### 1.3 图片导出存在 SSRF/信息泄露与挂起风险 ✅ P0 已落地

**文件**：`src/lib/exportDocx.ts`

**问题描述**：`fetchImage` 直接对任意 URL 发起 `fetch(url, { mode: 'cors' })`，无协议校验、无超时、无域名白名单。可被用于请求内网服务、`file://` 协议或恶意地址，且导出可能永久挂起。

**建议**：校验 URL 协议为 `http/https`，增加 `AbortSignal` 超时，必要时加域名白名单；对失败 URL 给出明确提示而非静默空白。

**落地状态**：✅ 已落地 - 已改用 `fetchSafe.ts` 的 `fetchImageBuffer`，含协议校验（仅限 http/https）与 30s 超时。

---

### 1.4 用户输入直接拼入 CSS ✅ P1 已落地

**文件**：`src/modes/document/paged/pagedPageCss.ts`

**问题描述**：页眉页脚文字直接拼入 CSS `content`，目前仅转义 `"` 与 `\`，未处理换行、控制字符，可能生成非法 CSS 或注入。

**建议**：增加对 `\n`、`\r`、`\x00` 等字符的过滤或转义；对图片 `size` 参数也做 CSS length 模式校验。

**落地状态**：✅ 已落地 - 已提供 `cssString()` 函数，转义 `\` 与 `"`；footer 文本分片也经转义处理。

---

### 1.5 ObjectURL 释放时间过短 ⚠️ P0 未完全关闭

**文件**：`src/lib/exportImage.ts`

**问题描述**：`downloadBlob` 用 `setTimeout(..., 1000)` 后立刻 `revokeObjectURL`，大文件在慢速浏览器/移动端可能还未开始下载就被释放。

**建议**：改为 `a.click()` 后延迟 10000 ms 释放，或监听 `visibilitychange` 再释放；失败时兜底 `navigator.msSaveOrOpenBlob`。

**落地状态**：❌ 尚未落地

## 二、状态管理与应用架构

### 2.1 单一巨型 Store 持久化性能瓶颈 ❌ P1 未落地

**文件**：`src/lib/store.ts:72-118, 307-330`

**问题描述**：所有状态（四份大文本、主题、图床、自定义指令）都塞进一个 Zustand store 并 `persist` 到 `localStorage`。每次按键都会触发整 store 序列化与写入，长文档下会阻塞主线程，且容易超过 5~10 MB 配额。

**建议**：
- 拆分为多个 store：如 `useAppStore`（主题/UI）、`useContentStore`（各模式文本）、`useInstructionStore`（自定义指令）。
- 大文本改用 IndexedDB 自定义 storage，或对 `localStorage` 写入做 throttle（如 1000 ms）。
- `colors` 是由 accent 推导出的运行时值，应从 `partialize` 中移除，避免老用户拿到旧值。

**落地状态**：❌ 尚未落地

---

### 2.2 模块级副作用与初始化时序问题 ✅ P1 已落地

**文件**：`src/lib/store.ts:184-187, 331-336`

**问题描述**：`getInitialStateFromLegacyKeys()` 与 `applyCssVars()` 在模块顶层运行，早于 React 生命周期。若后续做 SSR、预渲染或测试会很难控制，也可能造成首屏闪色。

**建议**：把迁移和 CSS 变量设置移到 `onRehydrateStorage` 或一次性的 `useInitializeApp()` hook 中，不要在模块加载时直接操作 `document`/`localStorage`。

**落地状态**：✅ 已落地 - `applyCssVars` 与迁移逻辑已移入 `onRehydrateStorage`，不再在模块顶层操作 DOM。

---

### 2.3 ErrorBoundary 不会随模式切换重置 ✅ P1 已落地

**文件**：`src/App.tsx:149`

**问题描述**：`ErrorBoundary` 包住了所有模式组件，但没有提供 `resetErrorBoundary` 或 `key`。如果 A4 文档模式崩溃，用户切换到长图文模式时错误状态仍在。

**建议**：给 `ErrorBoundary` 加 `key={mode}`，或在 `setMode` 时调用 reset。

**落地状态**：✅ 已落地 - 已添加 `key={mode}`，切换模式时自动重置错误状态。

**建议**：抽出通用 `ModeLayout` 组件，统一处理响应式布局与视图切换，各模式只关注自身预览与工具栏。

## 三、渲染引擎

### 3.1 parseMarkdown 是超大单体函数 ✅ P1 已落地

**文件**：`src/engine/utils/markdownParser.ts`

**问题描述**：所有 block 类型处理逻辑堆在一个 `while` 循环里，扩展新类型或修复单类 bug 都需要改动同一文件，单测难以针对单个 block 类型。

**建议**：基于已有的 `classifyBlock` 建立 block renderer 注册表（`Record<BlockKind, BlockRenderer>`），每种 block 拆成独立 renderer，`parseMarkdown` 只负责 front-matter、行遍历、dispatch 和结果拼接。

**落地状态**：✅ 已落地 - 已引入 `blockRenderRegistry.ts`，20+ 独立 renderer 通过注册表 dispatch。

---

### 3.2 代码块内的公式与脚注被错误替换 ❌ P1 未落地

**文件**：`src/engine/utils/math.ts`、`src/engine/utils/markdownParser.ts:260-275`

**问题描述**：`extractMath` 与脚注替换在整篇文本上进行，未排除代码围栏和行内代码，导致代码里的 `$$...$$` 或 `[text](url)` 被错误渲染为公式或 `__FN_` 占位符。

**建议**：统一实现「受保护区域」机制：在公式/脚注处理前先把代码块与行内代码用占位符跳过，渲染完成后再还原。

**建议**：增加未闭合容错：达到 EOF 时按普通段落渲染并给出 warning/Toast；或限制最大扫描行数，超过阈值自动截断降级。

### 3.6 大文档渲染性能与 Paged.js 全量重排 ❌ P2 未落地

**文件**：`src/modes/document/DocumentMode.tsx`、`src/modes/document/paged/usePagedPreview.ts`

## 四、导出功能

### 4.1 截图与 PDF 页面尺寸不同步 ✅ P1 已落地

**文件**：`src/lib/exportImage.ts`、`src/lib/exportPdf.ts`

**问题描述**：`captureElementInIframeToBlob` 截图时临时把 iframe 撑到内容尺寸，截图完成后才恢复样式；但 `exportPdf.ts` 在截图之后读取 `pageNodes[i].offsetWidth/offsetHeight`，此时样式已恢复，PDF 页面尺寸可能与实际截图不一致。

**建议**：让 `captureElementInIframeToBlob` 返回 `{ blob, width, height }`，调用方直接使用返回值创建 PDF 页面。

**落地状态**：✅ 已落地 - 直接复用 `captureElementInIframeToBlob` 返回的 `{ blob, width, height }`，尺寸同源。

---

### 4.2 公式重复渲染与行内代码冲突 ❌ P1 未落地

**文件**：`src/lib/exportDocx.ts`

**问题描述**：`convertParagraph` / `convertList` 各自调用 `renderAllFormulas`，块级公式还要再 `renderFormula`；且 `preprocessFormulas` 把 `$...$` 替换为反引号代码，若用户原本就有同名行内代码会被错误渲染为公式。

**建议**：建立文档级 `FormulaCache`，`exportToDocx` 开始时统一扫描并渲染所有公式，段落转换只读取缓存；预处理跳过已有反引号代码区域，或使用不会冲突的占位符。

**落地状态**：❌ 尚未落地

---

### 4.3 导出链路缺少取消与超时机制 ❌ P2 未落地

**文件**：`src/lib/useExportAction.ts`、`src/lib/exportPdf.ts`

**问题描述**：大文档导出期间用户无法中断；`useExportAction` 在组件卸载后仍可能 `setState`。

**建议**：导出动作接入 `AbortController`，支持中断并清理样式恢复；用 mounted ref 在 finally 中判断组件是否仍挂载。

**落地状态**：❌ 尚未落地

### 4.4 ZIP 内重名文件覆盖 ❌ P2 未落地

**建议**：打包前检测重名，自动追加 `-1`、`-2` 等序号；fallback 循环用 try/catch 包裹单个下载。

## 五、图片与资源管理

### 5.1 本地图片 URL 缓存无上限 ⚠️ P1 部分落地

**文件**：`src/lib/editor/imageStorage.ts`

**问题描述**：`localImageUrls` 是全局对象，永不清理；`resolveImageUrl` 还把 Blob 转成 base64 Data URL 缓存，内存占用高。

**建议**：
- 改为 LRU 缓存，超出数量时调用 `URL.revokeObjectURL`。
- 优先使用 `URL.createObjectURL` 代替 base64 Data URL，减少 Markdown 文本膨胀。
- 在文档切换、图片删除、组件卸载时清理无用 Object URL。

**落地状态**：⚠️ 部分落地 - IndexedDB 存储和 `compileMarkdownImages` 已有测试覆盖，但 `localImageUrls` 全局缓存仍无上限，Blob 转 base64 Data URL 的内存优化也未实施。

---

### 5.2 图片压缩与上传校验不足 ❌ P2 未落地

**文件**：`src/lib/editor/imageStorage.ts`、`src/lib/useImageUpload.ts`

**问题描述**：`compressImage` 固定输出 JPEG，PNG 透明通道会丢失；上传缺少文件大小/类型校验；`uploadImageFile` 用原文件名但 MIME 已变，扩展名与 MIME 不匹配。

**建议**：
- 根据原类型选择输出格式（PNG 保留透明）。
- 增加 `accept="image/*"`、`maxSize`、图片头校验。
- 按压缩后 MIME 修正扩展名（jpg/png/webp）。
- Sm.ms 响应做 `typeof url === 'string'` 校验。

**落地状态**：❌ 尚未落地

**问题描述**：`target: ES2021` 与 Vite 现代构建目标略有脱节；`noUnusedLocals/Parameters` 被关闭；模块级副作用未受约束。

**建议**：评估是否升级到 `ES2022`；开启 `noUnusedLocals` 并配合 lint 自动修复；将 store 初始化改为懒加载/Hook 初始化。

**落地状态**：❌ 尚未落地

---

### 6.4 子项目与根项目关系混乱 ❌ P2 未落地

**问题描述**：仓库根目录同时包含 `html-anything/`、`r-markdown/` 与 `markdown2view` 本身，顶层 `package.json` 未配置 workspace，但子目录又有自己的 `package.json`/`pnpm-workspace.yaml`。ESLint 直接忽略这些目录，容易造成依赖、脚本、文档的维护盲区。

**建议**：明确三者关系：若为参考/归档，应移到 `vendor/` 或子模块；若仍需维护，应配置 pnpm workspace 与统一 lint/typecheck 入口。

**落地状态**：❌ 尚未落地

## 七、剩余工作规划建议

基于当前落地状态，建议将剩余工作按「风险/收益」与「实施成本」重新排序，分三阶段推进：

### 第一阶段：关闭 P0 安全漏洞（1 周）

- 将云存储 SecretKey 移出前端 persist，改用后端代理或 STS 临时凭证方案
- 评估并引入 DOMPurify 替代自研 sanitizer，或强化自研方案测试覆盖
- 延长 ObjectURL 释放时间至 10000ms，增加 visibilitychange 兜底

### 第二阶段：补齐架构与核心正确性（2~3 周）

- 拆分巨型 Zustand store，大文本持久化改用 IndexedDB + throttle
- 实现「受保护区域」机制，修复代码块内公式/脚注被污染问题
- 自定义标签未闭合容错：EOF 降级 + 扫描行数限制
- DOCX 公式缓存：统一扫描渲染，消除重复计算与行内代码冲突
- 抽取 ModeLayout 组件，消除四个模式组件的重复布局代码

### 第三阶段：性能、工程化与测试补齐（2~4 周）

- 大文档自适应防抖与 memo 拆分，降低 Paged.js 全量重排频率
- 本地图片 LRU 缓存与 Object URL 生命周期管理
- 补齐 PDF/图片/ZIP 导出链路测试与 malformed 回归测试
- 收紧 ESLint 规则，补全浏览器全局对象类型定义
- 集中排版 token，逐步消除 magic number 硬编码
