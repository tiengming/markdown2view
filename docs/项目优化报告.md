## 一、安全与合规

### 1.1 云存储密钥暴露在客户端并被持久化 ✅ P0 已落地

**文件**：`src/lib/store.ts:61-66` 与 `src/lib/editor/imageStorage.ts`

**问题描述**：`ImageHostConfig` 直接在前端保存 OSS 的 `accessKeySecret` 与 COS 的 `SecretKey`，且通过 Zustand persist 写入 `localStorage`。只要用户打开 DevTools 或遭遇 XSS，密钥即可被读取，属于生产环境不可接受的风险。

**建议**：上传能力改为后端代理或临时预签名 URL 方案；前端只保留不敏感的配置项（如 bucket、region）。如必须保留前端上传，也应由后端签发限时 STS 临时凭证，并设置最小权限策略。

**落地状态**：✅ 已落地（纯前端方案）- 采用「分层」策略平衡安全与便捷：
- **密钥默认不落盘**：`store.ts` 的 `partialize` 通过 `stripImageHostSecrets()` 剥离 AK/SK/Token，仅持久化 `activeType` 与 `region/bucket` 等非敏感字段；密钥仅存在于当前会话内存，刷新即清除。
- **可选加密记忆**：新增 `src/lib/secureVault.ts`，基于浏览器原生 Web Crypto（PBKDF2-SHA256 派生 + AES-GCM-256）以用户口令加密密钥后存入 localStorage；`SettingsModal` 提供「记住密钥」开关、口令输入与解锁横幅。口令仅用于本地加密、不上传，遗忘无法恢复。
- **友好降级**：`uploadImageFile` 在密钥缺失/未解锁时给出明确提示（引导去设置解锁或填写），不再静默回退或抛出晦涩的 SDK 错误。
- **进阶方案（文档建议）**：对安全性要求更高者，可使用最小权限子账号（仅写入单 bucket）或自部署 Serverless 签名器签发 STS 临时凭证。

### 1.2 AI 返回 HTML 存在 XSS 风险 ⚠️ P0 部分落地

**文件**：`src/lib/extractHtml.ts`、`src/lib/htmlSanitizer.ts`

**问题描述**：`extractHtml` 对 AI 输出几乎不做净化：只要以 `<` 开头就直接返回，并注入 iframe 渲染。恶意脚本、事件处理器或 `javascript:` 链接均可能在预览/导出阶段执行。

**建议**：
- 引入 `DOMPurify` 对 HTML 做净化，默认禁用 `script` 与事件属性。
- 若业务确实需要执行脚本，使用 `iframe[srcdoc]` 配合 `sandbox="allow-scripts"`，并设置严格 CSP。
- `previewHtml` 中的 link/style/script 注入改用 `DOMParser` + 程序化 DOM 操作，避免字符串正则切分带来的边界错误。

**落地状态**：✅ 已评估 - 自由画布 HTML 依赖完整 CSS 表达能力（变量、动画、Flex/Grid、SVG 等），不适合采用会大范围过滤 CSS 的严格白名单净化方案。当前采用「高风险黑名单 + iframe sandbox + AI 指令约束 + 测试覆盖」策略：

- **黑名单净化**：`htmlSanitizer.ts` 基于 DOMParser，拦截 `<script>`、事件处理器属性（`onclick` 等）、`javascript:` / 可执行 `data:` 协议、CSS `expression` / `behavior` / `@import`；保留所有标签、`class`、`style`、`data-*`、SVG、CSS 变量等。
- **iframe sandbox**：`HtmlSandbox` 默认 `sandbox="allow-same-origin"`，用户可手动开启 `allow-scripts` 交互演示，见 `HtmlMode.tsx`。
- **AI 指令同步**：HTML 设计提示词明确禁止 `<script>`、事件属性、危险 URL、`@import`、`expression`、`behavior`，并要求样式内联在 `<style>` 中，见 `designPrompts/utils.ts`。
- **测试覆盖**：`htmlSanitizer.test.ts` 23 个测试全部通过，覆盖保留样式、移除危险向量、SVG、`data:image/*`、复杂页面结构等关键用例。

**后续可选方向**：DOMPurify 可作为可选评估项，但必须使用兼容自由画布的宽松配置，并验证不会移除 `<style>`、内联样式、CSS 变量、动画、Flex/Grid、SVG 等关键能力。

### 1.3 图片导出存在 SSRF/信息泄露与挂起风险 ✅ P0 已落地

**文件**：`src/lib/exportDocx.ts`

**问题描述**：`fetchImage` 直接对任意 URL 发起 `fetch(url, { mode: 'cors' })`，无协议校验、无超时、无域名白名单。可被用于请求内网服务、`file://` 协议或恶意地址，且导出可能永久挂起。

**建议**：校验 URL 协议为 `http/https`，增加 `AbortSignal` 超时，必要时加域名白名单；对失败 URL 给出明确提示而非静默空白。

**落地状态**：✅ 已落地 - 已改用 `fetchSafe.ts` 的 `fetchImageBuffer`，含协议校验（仅限 http/https）与 30s 超时。

### 1.5 ObjectURL 释放时间过短 ✅ P0 已落地

**文件**：`src/lib/exportImage.ts`

**问题描述**：`downloadBlob` 用 `setTimeout(..., 1000)` 后立刻 `revokeObjectURL`，大文件在慢速浏览器/移动端可能还未开始下载就被释放。

**建议**：改为 `a.click()` 后延迟 10000 ms 释放，或监听 `visibilitychange` 再释放；失败时兜底 `navigator.msSaveOrOpenBlob`。

**落地状态**：✅ 已落地 - `downloadBlob` 已将 ObjectURL 释放延迟从 1000ms 调整为 10000ms，并通过 `visibilitychange` 在页面切后台时提前释放；同时增加 `navigator.msSaveOrOpenBlob` 兼容分支，覆盖旧版浏览器下载兜底。

## 二、状态管理与应用架构

### 2.1 单一巨型 Store 持久化性能瓶颈 ✅ P1 已落地

**文件**：`src/lib/appStore.ts`、`src/lib/contentStore.ts`、`src/lib/idbStorage.ts`、`src/App.tsx`

**问题描述**：所有状态（四份大文本、主题、图床、自定义指令）都塞进一个 Zustand store 并 `persist` 到 `localStorage`。每次按键都会触发整 store 序列化与写入，长文档下会阻塞主线程，且容易超过 5~10 MB 配额。

**建议**：
- 拆分为多个 store：如 `useAppStore`（主题/UI）、`useContentStore`（各模式文本）、`useInstructionStore`（自定义指令）。
- 大文本改用 IndexedDB 自定义 storage，或对 `localStorage` 写入做 throttle（如 1000 ms）。
- `colors` 是由 accent 推导出的运行时值，应从 `partialize` 中移除，避免老用户拿到旧值。

**落地状态**：✅ 已落地
- 新增 `src/lib/idbStorage.ts`，实现 Zustand persist 兼容的 IndexedDB 自定义 storage，并带 1000ms 写入节流，合并同一 key 的连续写入；
- 拆分原 `useStore` 为 `useAppStore`（localStorage，存放主题/UI/字体/文档设置/图床/自定义指令）与 `useContentStore`（IndexedDB，存放四份大文本及 dirty/demoVersion），彻底避免大文本与 UI 状态互相触发序列化；
- `useAppStore` 的 `partialize` 移除派生的 `colors`，仅持久化 `accent`/`accentDark`，rehydrate 时重新计算，避免老用户拿到旧 token；
- `src/lib/store.ts` 保留 `useStore` 别名指向 `useAppStore` 并重新导出所有类型，现有组件无需立即全量迁移；
- `App.tsx` 改为分别订阅两个 store，并保留 document 模式恢复示例时的页眉页脚设置重置逻辑；
- 新增 `src/lib/idbStorage.test.ts`、`src/lib/contentStore.test.ts`、`src/lib/appStore.test.ts` 覆盖节流、大文本持久化、secret 剥离等核心行为。

## 四、导出功能

### 4.1 截图与 PDF 页面尺寸不同步 ✅ P1 已落地

**文件**：`src/lib/exportImage.ts`、`src/lib/exportPdf.ts`

**问题描述**：`captureElementInIframeToBlob` 截图时临时把 iframe 撑到内容尺寸，截图完成后才恢复样式；但 `exportPdf.ts` 在截图之后读取 `pageNodes[i].offsetWidth/offsetHeight`，此时样式已恢复，PDF 页面尺寸可能与实际截图不一致。

### 4.2 公式重复渲染与行内代码冲突 ✅ P1 已落地

**文件**：`src/lib/exportDocx.ts`

**问题描述**：`convertParagraph` / `convertList` 各自调用 `renderAllFormulas`，块级公式还要再 `renderFormula`；且 `preprocessFormulas` 把 `$...$` 替换为反引号代码，若用户原本就有同名行内代码会被错误渲染为公式。

**建议**：建立文档级 `FormulaCache`，`exportToDocx` 开始时统一扫描并渲染所有公式，段落转换只读取缓存；预处理跳过已有反引号代码区域，或使用不会冲突的占位符。

**落地状态**：✅ 已落地
- 新增 `FormulaCache` 类：`exportToDocx` → `buildDocument` 开始时统一扫描整篇文档的所有 `$...$` / `$$...$$` 公式并渲染，段落/列表转换仅通过 `formulaMap` 读取已渲染结果，不再逐块重复调用 MathJax。
- 公式预处理保护代码区域：`protectCodeRegions` / `restoreCodeRegions` 先将代码块与行内代码用 Unicode 私有占位符隔离，再执行公式替换，最后还原；避免用户原有的反引号代码 `$...$` 被误识别为公式。
- 块级公式检测：段落转换时通过正则识别 `` `__BLOCK_FORMULA_N__` `` 占位符，直接嵌入已渲染的 SVG/PNG，不再二次渲染。
- 测试覆盖：`src/lib/exportDocx.test.ts` 已覆盖 FormulaCache 占位符生成、代码区域保护还原、公式替换、代码块/行内代码内公式跳过等 7 个用例。

## 五、图片与资源管理

### 5.1 本地图片 URL 缓存无上限 ✅ P1 已落地

**文件**：`src/lib/editor/imageStorage.ts`

**问题描述**：`localImageUrls` 是全局对象，永不清理；`resolveImageUrl` 还把 Blob 转成 base64 Data URL 缓存，内存占用高。

**建议**：
- 改为 LRU 缓存，超出数量时调用 `URL.revokeObjectURL`。
- 优先使用 `URL.createObjectURL` 代替 base64 Data URL，减少 Markdown 文本膨胀。
- 在文档切换、图片删除、组件卸载时清理无用 Object URL。

**落地状态**：✅ 已落地 - `imageStorage.ts` 已将本地图片预览缓存改为 ObjectURL + LRU 策略，最多保留 64 个 URL，超出时自动调用 `URL.revokeObjectURL` 释放最旧项；`resolveImageUrl` 不再把 Blob 长期缓存为 base64 Data URL。新增 `revokeImageUrl()`、`clearLocalImageUrlCache()`、`pruneLocalImageUrlCache()`，并在 `preloadImagesFromMarkdown()` 中按当前 Markdown 引用清理无用 URL；`CodeEditor` 会随 Markdown 内容变化预加载/修剪图片，并在组件卸载时清空缓存。`imageStorage.test.ts` 已补充 ObjectURL 缓存复用、按 id 释放、未使用项修剪与 LRU 淘汰测试。

### 5.2 图片压缩与上传校验不足 ✅ P2 已落地

**文件**：`src/lib/editor/imageStorage.ts`、`src/lib/useImageUpload.ts`

**问题描述**：`compressImage` 固定输出 JPEG，PNG 透明通道会丢失；上传缺少文件大小/类型校验；`uploadImageFile` 用原文件名但 MIME 已变，扩展名与 MIME 不匹配。

**建议**：

- 增加 `accept="image/*"`、`maxSize`、图片头校验。
- 按压缩后 MIME 修正扩展名（jpg/png/webp）。
- Sm.ms 响应做 `typeof url === 'string'` 校验。

**落地状态**：✅ 已落地 - `compressImage` 新增 `outputType` 参数，根据原类型选择输出格式（PNG 保留透明通道）；`uploadImageFile` 前置调用 `validateImageFile`，校验文件类型（仅支持 JPG/PNG/WebP）、大小（默认 10MB）和文件头一致性；`createImageUploadFile` 按压缩后 MIME 修正扩展名；`generateImageFilename` 也按 MIME 生成扩展名；`uploadToSmMs` 对响应 URL 做严格 `typeof url === 'string'` 校验。`imageStorage.test.ts` 已补充图片类型/大小校验、非图片拒绝、超大文件拒绝、扩展名修正测试。
