分页拆分引擎技术验证报告

第一章 背景与单段中部分页验证

本页之后是一段刻意写得很长的段落，用于验证 Paged.js 能否在单个段落内部按视觉行断开并续接到下一页——这正是"测量后打包"方案中最难手搓的部分。在传统的整块分页策略里，一个超长段落如果在当前页放不下，就会被整体推到下一页，从而在页面底部留下大片空白；而基于 CSS Paged Media 的流式断点引擎会自动在合适的视觉行处断开，最大化利用版面空间。

在实际的报告与公文排版中，长段落极其常见：政策解读、技术说明、案情陈述、研究综述往往一段就有数百字。如果分页引擎不能在段落中部断开，排版结果就会出现明显的"莫名换页留白"，既浪费纸张也不专业。下面这一段会被刻意拉长到跨越整页边界，请重点观察它在页面底部的断开位置是否自然、续接到下一页后是否从正确的位置继续，并且断开处不应出现文字被截断或重叠的情况。这一段开始重复填充内容以确保跨页：分页是把连续内容按页面物理高度切分为若干页的过程，其核心难点在于"在哪里断开"。对于离散块（标题、图片、表格行）断点是天然的；但对于连续文本，断点存在于任意两个视觉行之间，必须由排版引擎结合字体、字号、行高、容器宽度计算出每一行的实际位置。Paged.js 通过在渲染流中检测溢出节点与字符偏移（breakToken）来定位断点，从而实现段落中部的精确分页。这一段继续重复以保证它一定会跨越页面边界：在浏览器中，我们可以利用 Range 对象逐步扩展并读取 getClientRects 的矩形数量来推断视觉行边界，这也是 Paged.js 内部寻找文本断点的原理之一。但若由我们自行实现，则需要处理二分查找优化、Safari 空白折叠差异、以及拆分后 DOM 重排带来的再测量问题，工程量与维护成本都不低。因此本 POC 旨在验证：直接复用 Paged.js 是否能以更低成本获得同等甚至更好的分页质量。继续填充以跨页：理想的结果是，这一长段在页面底部自然断行、在下一页顶部无缝续接，全程没有文字截断、没有重叠、没有多余留白，并且页眉页脚与页码在新页上正确显示。

第二章 列表、引用与代码块

以下验证常见块级元素在跨页时的表现。列表项之间应可在页面边界自然断开，单个列表项内部尽量保持完整。

  • 列表项一：分页引擎应允许列表在项与项之间跨页。
  • 列表项二：Paged.js 默认在合适位置断开，配合 break-inside 控制单项完整性。
  • 列表项三：相比手动方案，无需把每个列表项预先拆成独立块。
  • 列表项四：有序与无序列表的编号/项目符号应在续接页保持正确。
引用块验证：引用块跨页时应保留左侧色条与背景内边距，续接页的引用样式不应丢失。这是手动方案中需要在每个续接片段重新补回 > 包裹的麻烦点，而 CSS 引擎可自动延续容器样式。 
// 代码块（<10 行，按用户决策保持整体不拆）
function paginate(blocks, pageHeight) {
  const pages = []
  let cur = [], used = 0
  for (const b of blocks) {
    if (used + b.h > pageHeight) { pages.push(cur); cur = []; used = 0 }
    cur.push(b); used += b.h
  }
  return pages
}

【内联样式块验证】这一段使用与渲染引擎一致的内联样式（style="margin:...;line-height:1.78"）直接写在元素上，用于验证 Paged.js 不依赖外部类名也能正确分页带内联样式的内容。我们的引擎输出的就是这种逐行 <section><p style="..."> 结构，因此此项通过即说明可直接喂入引擎产物。
